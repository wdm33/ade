## Conventions

- **Source of rules.** The canonical, append-only source is the invariant
  registry at `.idd-config.json` `invariant_registry`
  (`docs/ade-invariant-registry.toml`). The registry was renamed from
  `constitution_registry.toml` in the post-Phase-4-N-D IDD-canonicalization
  delta (`ci_check_constitution_coverage.sh` was repointed in `39865f6`).
  Every `[[rules]]` entry whose `status` is not explicitly retired appears
  below. None are deprecated at HEAD.
- **Rule IDs** are stable across regenerations and immutable per
  registry doctrine. They are reproduced exactly as the registry holds them.
- **Families.** The registry groups rules under five family-letter
  prefixes:

  | Prefix | Family |
  |---|---|
  | `T-` | True invariants (project constitution §2) |
  | `DC-` | Derived Cardano compatibility invariants (§3) |
  | `CN-` | Classification-table invariants (`classification_table.md`) |
  | `RO-` | Release obligations (§4a) |
  | `OP-` | Operational invariants (§4b) |

  Within each family, level-3 sub-headings hold one rule each. The
  sub-section ID *is* the rule ID. Within a family, ordering follows
  the registry file order.
- **Four-cell contract.** Every rule carries the canonical four-row table:
  Source, Requirement, Code, Tests, CI. Where the registry has populated
  `code_locus`, `tests`, `ci_script`, those fields drive the row. A row
  marked `_(no enforcement found — gap)_` corresponds to a registry
  entry whose `status` is `declared` (no enforcement promised) or to a
  partial entry that has not yet pointed at a code/test/CI artifact.
- **CI script BLUE-scope subtlety.** Five CI scripts
  (`ci_check_module_headers.sh`, `ci_check_no_semantic_cfg.sh`,
  `ci_check_no_signing_in_blue.sh`, `ci_check_hash_uses_wire_bytes.sh`,
  `ci_check_ingress_chokepoints.sh`) scan only the narrow
  `{ade_codec, ade_types, ade_crypto, ade_core}` BLUE list.
  `ci_check_dependency_boundary.sh` covers five (adds `ade_ledger`).
  `ci_check_forbidden_patterns.sh` covers all six (adds `ade_plutus`).
  `.idd-config.json` declares six BLUE crates. Rules whose enforcement
  depends on a narrow-scope script have `_(scope gap)_` flagged on
  the CI row when their requirement also applies to `ade_ledger` or
  `ade_plutus`. See CODEMAP "Subtlety in BLUE scope."

## Cross-references and consistency warnings

These warnings are surfaced for the parent's downstream review; they do
not block doc generation.

### A. Five CI scripts use a narrow 4-crate BLUE list

CODEMAP "Subtlety in BLUE scope" identifies the largest enforcement gap
in the workspace. `.idd-config.json` declares six BLUE crates
(`ade_codec`, `ade_types`, `ade_crypto`, `ade_core`, `ade_ledger`,
`ade_plutus`). Five CI scripts hard-code a four-crate list
(`ade_codec`, `ade_types`, `ade_crypto`, `ade_core`):

- `ci/ci_check_module_headers.sh`
- `ci/ci_check_no_semantic_cfg.sh`
- `ci/ci_check_no_signing_in_blue.sh`
- `ci/ci_check_hash_uses_wire_bytes.sh`
- `ci/ci_check_ingress_chokepoints.sh`

A sixth (`ci/ci_check_dependency_boundary.sh`) covers five
(adds `ade_ledger`, omits `ade_plutus`). Only
`ci/ci_check_forbidden_patterns.sh` matches the declared six-crate BLUE
list.

**Affected rules — narrow CI is the only enforcement and `ade_ledger`/`ade_plutus` are not scanned:**

| Rule | Narrow-scope CI | Risk on `ade_ledger`/`ade_plutus` |
|---|---|---|
| `T-ENC-01` | `ci_check_hash_uses_wire_bytes.sh` | A `hash(canonical_bytes)` call inside `ade_ledger` or `ade_plutus` would not fail CI; `ade_ledger` is the largest source of hashing and is therefore the highest-risk crate. |
| `T-BUILD-01`, `CN-BUILD-01`, `CN-BUILD-02` | `ci_check_no_semantic_cfg.sh` | A `#[cfg(feature = ...)]` added to `ade_ledger` (or `ade_plutus`) would not fail CI. |
| `T-KEY-01`, `DC-CRYPTO-02`, `CN-CRYPTO-01` | `ci_check_no_signing_in_blue.sh` | A `SigningKey` import in `ade_ledger` or `ade_plutus` would not fail CI. |
| `T-INGRESS-01`, `DC-INGRESS-01`, `CN-WIRE-06` | `ci_check_ingress_chokepoints.sh` | A direct `PreservedCbor::new` call from `ade_ledger` or `ade_plutus` (if visibility were relaxed) would not fail CI — only the `pub(crate)` visibility in `ade_codec` prevents it today, so the *type-level* enforcement remains intact even though the *CI-level* enforcement does not. |
| (banner/headers) | `ci_check_module_headers.sh` | New `.rs` files under `ade_ledger`/`ade_plutus` can ship without the `// Core Contract:` banner without failing CI. |
| `DC-CBOR-02` | `ci_check_hash_uses_wire_bytes.sh` | Same as `T-ENC-01`. |
| `T-BOUND-02` (cross-ref) | `ci_check_dependency_boundary.sh` (5-crate) | A future `ade_plutus`-on-`ade_runtime` dependency would not fail CI. |

**Recommended close (not done by this generator):** unify the BLUE list
in a shared shell variable read by every BLUE-scoped script, or pass
the list from `.idd-config.json` `core_paths`.
