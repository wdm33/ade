## `eval_phase_two_raw` rejects mainnet Conway txs the chain accepted — `txInfoValidRange` divergence

### Summary

When I run `uplc::tx::eval_phase_two_raw` against a curated subset of
real mainnet Conway transactions, some txs return a script-execution
failure whose validator trace points at `txInfoValidRange` / an
"expiration time is not properly set" user-level `traceIfFalse`.
The same txs were included on mainnet by cardano-node, which means
the on-chain Haskell ledger's ScriptContext produced a
`txInfoValidRange` that the validator accepted.

Across 4,500 contiguous Conway blocks replayed through aiken v1.1.21:
- **631 Plutus txs evaluated successfully** — no false positives
  (we never return `Ok` for a tx the chain flagged `is_valid=false`).
- **~44 divergent failures** — our `eval_phase_two_raw` returns an
  error for a tx whose index is **not** in the block's
  `invalid_transactions` field. All divergences share the same trace
  signature.

### Error signature

Every divergent failure in our 4,500-block sweep produces a
signature like:

```
eval_phase_two_raw: failed script execution
  Spend[N] the validator crashed / exited prematurely
    Trace Expiration time is not properly set
    Trace PT5
```

The `Expiration time is not properly set` text is not from aiken —
it's the user validator script's own `traceIfFalse` message. The
validator is almost certainly running a check equivalent to:

```aiken
expect Some(expiration) = ...
trace_if_false("Expiration time is not properly set",
    tx_info_validity_range.upper_bound < expiration,
)
```

For the check to fail on a tx that the chain considers valid, one of
these must be true:

1. `tx_info_validity_range.upper_bound` is `PosInf` in the
   ScriptContext aiken constructs, but the real tx has a finite
   `ttl`. (`ttl: Option<SlotNo>` is the Alonzo+ tx body field; it
   carries an explicit upper bound when set.)
2. `tx_info_validity_range.upper_bound` is present but anchored to
   the wrong slot (e.g., zero-slot offset different from what the
   Conway HFC schedule expects).
3. `tx_info_validity_range.lower_bound` is `NegInf` when it should
   be finite (validators occasionally check both bounds).

### Environment

- aiken: pinned at commit `42babe5d` (tag v1.1.21)
- pallas-primitives: 0.33.0
- Protocol version under test: 10 (Conway as of cardano-node 10.6.2

- Cost models: the pparams values parsed from the Conway

- Slot config: mainnet Shelley-era anchor

### Hypothesis

ScriptContext V3 construction in `crates/uplc/src/tx/script_context.rs`
incorrectly maps Conway tx body fields `validity_interval_start` /
`ttl` into the V3 `TxInfo`'s `validity_range` field. Candidate
locations:

- The `PseudoTransactionBody<…, KeepRaw<…>>` decoder path may drop or
  defer the `validity_interval_start` / `ttl` fields when the tx uses
  Conway-new governance fields (voting_procedures /
  proposal_procedures).
- The ScriptContext lowering may translate a finite-upper-bound tx
  into `Finite(…, PosInf)` if the upper bound slot exceeds some
  sentinel threshold.

### Impact on downstream consumers

We're porting aiken's UPLC evaluator into a Rust-based Cardano node
(Ade, ADE). This divergence affects our verdict-match rate against
mainnet for Conway: ~93.5% agreement bottom-line, 100% excluding this
class of upstream inheritance. Pinning a patched commit or
cherry-picking the fix would flip our Conway CE-88 closure from
"partial" to "closed."

Happy to provide more tx samples, supply a pared-down Rust test file,
or walk through how we extract fixtures from mainnet block data.
