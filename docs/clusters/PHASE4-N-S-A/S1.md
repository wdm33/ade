## §2 OQ audit resolutions

### OQ1 — `verify_header_kes` consumes header_body_bytes

**Confirmed at plan time.** `ade_core::consensus::kes_check::verify_header_kes`
line 106 calls `verify_kes(&kes_vk, period, &kes.kes_signature, &kes.header_body_bytes)`.
The KES signing input is the `header_body_bytes` field of
`HeaderKes` (in `ade_core::consensus::header_summary`,
line 57: "The exact header-body CBOR bytes the KES
signature was produced over").

→ A2's recipe MUST produce the same CBOR encoding of
`ShelleyHeaderBody` that the validator extracts as
`header_body_bytes`.

### OQ2 — `forge_block` construction sequence

**Audited.** `forge_block` currently accepts a
**pre-computed** `tick.kes_signature` field on
`ProducerTick`. It does NOT call any signing primitive
internally. So the current flow is:

```
external caller → builds KesSignature → ProducerTick → forge_block (BLUE)
```

The N-R-A `run_real_forge` step 3 calls
`shell.kes_sign_at(period, expected_vrf_input)` with a
placeholder — the KES signature in the resulting block
does NOT match what `header_body_bytes` would expect.

**A3 resolution (recommended):** thread a new bridge
function in `ade_runtime::producer::scheduler` (or a new
sibling module) that runs the 9-step sequence:

1. From `ProducerShell` + `ProducerTick` inputs, build the
   body buckets (currently inside `forge_block`).
2. Compute `body_hash` and `body_size`.
3. Call `unsigned_header_pre_image(...)` (A2's BLUE recipe)
   with the canonical inputs.
4. Call `producer_shell.kes_sign_header(period, &preimage)`.
5. Inject the resulting `KesSignature` into `ProducerTick`.
6. Call `forge_block(&tick)` (unchanged signature).

This minimizes disruption to `forge_block` while moving
the KES signature construction onto the canonical pre-image
path. **No placeholder/partial-block signing.**
