## Slice-Entry Proof Obligations (gating)

- **PO-1 (param index).** `dRepActivity` is the Conway protocol-parameter array
  field **immediately after** `dRepDeposit`. The GREEN loader currently reads gov
  params through `drep_deposit` via `read_uint_field(28)`
  (`snapshot_loader.rs:2565`). Confirm the next positional `read_uint_field`
  index against the real snapshot's pparams sub-map before wiring; do **not**
  assume `29` — pin it empirically against `corpus/snapshots/` Conway data.
- **PO-2 (placement).** `drep_activity` is a *protocol parameter*, not gov state.
  Its canonical home is `ConwayOnlyDepositParams` (the `Option`-gated
  Conway-only-params carrier; present-iff-Conway is exactly the required
  property), **not** `ConwayGovState`. Update the type's doc to reflect it now
  carries the DRep-activity param.
- **PO-3 (fingerprint honesty).** `ConwayOnlyDepositParams` is fingerprinted
  (`fingerprint.rs:243-248`, Conway-deposit tag). Adding `drep_activity` to the
  state struct (which derives `Eq`) requires adding it to the fingerprint —
  otherwise two distinct states fingerprint identically. This is a deliberate
  params-fingerprint migration: bump `FINGERPRINT_VERSION`, update the
  Conway-with-deposits golden, regenerate the `conway_epoch_508` boundary pin if
  the snapshot is present (pre-Conway pins unaffected — `None`).

## Hard Prohibitions
- No consumption of `GovCertEnv` in any apply/accumulation path (that is B5-S2/S3).
- No defaulting of `drep_activity` on the Conway path (absence is a fail-fast).
- No placing `drep_activity` in the shared `ProtocolParameters` (would make a
  Conway-only param representable in non-Conway states).
