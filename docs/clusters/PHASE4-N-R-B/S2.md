# Slice B3 — Per-peer dispatch wiring in produce_mode

**Cluster:** PHASE4-N-R-B.
**Combined naming:** the cluster plan's B3 deliverable lands as
file `S2.md` (B1+B2 combined under `S1.md`; this slice = B3 in
cluster-plan terms).
**Strengthens:** sets `CN-SNAPSHOT-01` + `CN-SNAPSHOT-02` up to
flip at B4 close. Replaces the previous `_ => {}` absorption in
`produce_mode::handle_listener_event`.
**Introduces:** `produce_mode::dispatch_server_frame_event` +
`produce_mode::ServerPeerStates` type alias.
**Dependencies:** PHASE4-N-R-B B1+B2 (HEAD `f5924cc`).

## §1 Intent

Replace the previous `_ => {}` arm in
`produce_mode::handle_listener_event` for server-side frame
events with explicit dispatch into the
`ade_runtime::network::n2n_server::dispatch_*` reducers. The
reducers advance per-peer state + compute response bytes; the
response bytes are not transmitted back to the peer yet
(transmit wire requires extending `MuxPump` with an outbound
relay — documented as N-R-C / future-cluster work).

## §2 Surface change

```rust
// New helper:
fn dispatch_server_frame_event(
    event: &OrchestratorEvent,
    peers_state: &mut ServerPeerStates,
    served_chain_view: &ServedChainView,
) -> Result<usize, &'static str>;

// handle_listener_event signature extended with peers_state +
// served_chain_view; the call site in run_produce_mode wires
// them.

type ServerPeerStates = BTreeMap<PeerId, PerPeerN2nServerState>;
```

`PeerConnected { role: DownstreamServer }` now inserts a
`PerPeerN2nServerState::new(chain_sync_version,
block_fetch_version)` into `peers_state`. `PeerDisconnected`
removes it. Server-side frame events look up the state, call
`dispatch_chain_sync_frame` or `dispatch_block_fetch_frame`,
and re-insert the updated state.

## §3 Honest scope

What B3 ships:

- Per-peer `PerPeerN2nServerState` map keyed by `PeerId`.
- Explicit dispatch into the existing N-G reducers
  (`dispatch_chain_sync_frame`, `dispatch_block_fetch_frame`).
- Snapshot reads via `served_chain_view.borrow()` — atomic
  read of the current `ServedChainSnapshot` (DQ-B1 watch
  semantics).
- No `_ => {}` absorption for server-side frame variants.

What B3 does NOT ship:

- Transmitting response bytes back to the peer. The reducer
  output is computed (response length tracked) but not
  written to the peer's TCP socket. The transmit wire
  requires extending `MuxPump` with an outbound-relay
  channel + a shared `PeerId → mpsc::Sender<Vec<u8>>` map.
  Documented as N-R-C work; the bounty-facing operator pass
  needs this closed before peers can fetch forged blocks.
- `BroadcastBlock → push_atomic` wiring (B4).
- Stash for `AcceptedBlock` tokens between `run_real_forge`
  and the `BroadcastBlock` arm (B4).

## §4 Tests

The N-Q smoke test (`produce_mode_starts_runs_three_slots_and_exits_via_max_slots`)
still passes — no peer connects during the 3-slot run, so the
new dispatch path is unused. B4 adds an integration test that
spawns a synthetic dialer peer + drives a frame exchange to
exercise the new dispatch path end-to-end.
