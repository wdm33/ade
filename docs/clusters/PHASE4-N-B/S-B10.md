## 3. Implementation Instruction (AI)

This slice composes everything. It introduces:

1. **A GREEN chain-selector orchestrator** in `ade_runtime` that
   threads N-A `ValidatedHeaderSummary` events and `RollBackRequest`s
   through BLUE.
2. **A stream-replay test harness** in `ade_testkit` that drives the
   orchestrator with a canonical ordered stream from a corpus file
   and asserts replay equivalence.
3. **A new crate `ade_core_interop`** mirroring N-A's `ade_network_
   interop` pattern: a RED binary that talks to a real cardano-node
   over N-A and asserts best-chain agreement for a sustained window.

The interop binary's release-evidence test is `#[ignore]` by default
(per S-A10 precedent) and run manually with `cargo test --release
-- --ignored` against a pinned-Docker cardano-node 10.6.2. The
evidence log lands at `docs/clusters/PHASE4-N-B/CE-N-B-6_<date>.log`.

Commit with the `Co-Authored-By: Claude <model+context>
<noreply@anthropic.com>` trailer.

## 4. Intent

Two intents:
1. (CE-N-B-5) Make replay-divergence in the consensus path
   impossible: a canonical stream replays byte-identically. This is
   the *cluster-wide* replay obligation.
2. (CE-N-B-6) Make divergence from a real cardano-node observable:
   if Ade's chain-selector disagrees with a paired cardano-node
   peer, the test fails loudly. This is the *interop* obligation.

## 9. Design Summary

### GREEN chain-selector orchestrator

```rust
// ade_runtime::consensus::chain_selector

use ade_core::consensus::candidate::{CandidateFragment, ChainSelectorState, TiebreakerView};
use ade_core::consensus::events::{BlockDistance, ChainEvent, Point, SecurityParam};
use ade_core::consensus::fork_choice::select_best_chain;
use ade_core::consensus::header_summary::{HeaderInput, ValidatedHeaderSummary};
use ade_core::consensus::header_validate::{validate_and_apply_header, HeaderApplied};
use ade_core::consensus::ledger_view::LedgerView;
use ade_core::consensus::era_schedule::EraSchedule;
use ade_core::consensus::praos_state::PraosChainDepState;
use ade_core::consensus::rollback::{apply_rollback, RollBackApplied, RollBackRequest};
use ade_core::consensus::errors::HeaderValidationError;

#[derive(Debug, Clone, PartialEq, Eq)]
pub enum StreamInput {
    HeaderArrival(HeaderInput),
    RollBack(RollBackRequest),
    EpochBoundary { new_epoch: ade_types::EpochNo },
}

#[derive(Debug, Clone, PartialEq, Eq)]
pub enum OrchestratorError {
    HeaderInvalid(HeaderValidationError),
}

/// Orchestrator state holds both PraosChainDepState and
/// ChainSelectorState. Mutable; updated step-by-step as inputs arrive.
#[derive(Debug, Clone, PartialEq, Eq)]
pub struct OrchestratorState {
    pub chain_dep:        PraosChainDepState,
    pub selector:         ChainSelectorState,
    /// Maintained for rollback replay convenience: list of
    /// (block_no, chain_dep_snapshot) at recent blocks. Pruned to
    /// last k entries.
    pub recent_snapshots: Vec<(ade_types::BlockNo, PraosChainDepState, TiebreakerView)>,
}

/// Process one stream input, return the emitted event(s) and the
/// new state. Pure but stateful via &mut self for ergonomics.
pub fn process_stream_input(
    state:        &mut OrchestratorState,
    input:        &StreamInput,
    ledger_view:  &dyn LedgerView,
    era_schedule: &EraSchedule,
) -> Result<ChainEvent, OrchestratorError>;
```

Algorithm:
- `HeaderArrival(h)`:
  1. Call `validate_and_apply_header(&state.chain_dep, &h,
     ledger_view, era_schedule)` → `HeaderApplied`.
  2. Update `state.chain_dep = applied.new_state`.
  3. Construct a single-header `CandidateFragment` rooted at
     `state.selector.current_tip` with rollback_depth = 0 and the
     just-validated header.
  4. Call `select_best_chain(&state.selector, &[candidate])`.
  5. On `ChainEvent::ChainSelected` advance the selector state and
     push a snapshot to `recent_snapshots`. Trim to last k entries.
  6. Return the event.
- `RollBack(req)`:
  1. Look up `rolled_back_chain_dep` from `recent_snapshots` by
     `req.to_block_no`. If absent → `ChainEvent::Rejected {
     reason: ChainSelectionReject::ForkBeforeImmutableTip }` (or
     `ExceededRollback` — whichever is structurally correct; in
     practice "snapshot not available locally" means it's older
     than k blocks → `ExceededRollback`).
  2. Look up `rolled_back_tiebreaker` from the same entry.
  3. Call `apply_rollback`.
  4. On success: update both state and chain-dep from
     `rolled_back_chain_dep`. Pop snapshots newer than `to_block_no`.
- `EpochBoundary { new_epoch }`:
  1. Apply `nonce::apply_nonce_input(&state.chain_dep,
     &NonceInput::EpochBoundary { new_epoch, last_block_of_prev_
     epoch: state.chain_dep.last_epoch_block })`.
  2. Update `state.chain_dep`. No `ChainEvent` is emitted for an
     epoch boundary alone — return a synthetic `RolledForward`?
     Actually no — emit nothing or define a new event. **Decision**:
     emit `ChainEvent::RolledForward { from: current_tip, to:
     current_tip }` to indicate "no chain change, just internal
     evolution". (Or: define a new variant. Since events are closed,
     and the cluster shipped 5 variants, prefer `RolledForward` as
     a no-op transition signal. Implementer's call: documented
     either way.)
  - **Better**: return `Option<ChainEvent>` from
    `process_stream_input` and emit `None` for epoch boundary. Update
    signature accordingly:
    ```rust
    pub fn process_stream_input(...) -> Result<Option<ChainEvent>, OrchestratorError>;
    ```
  - The stream replay test asserts that the orchestrator's *full
    history of emitted events* (filtering `None`) is identical across
    runs.

### Stream corpus

`corpus/consensus/stream/synthetic_session.json` — a hand-crafted
synthetic session that exercises:
1. Genesis → 5 valid headers
2. Rollback to block 3 (depth 2)
3. 4 valid headers on the new branch (taking block_no past the old
   tip)
4. Header that triggers an op-cert regression → rejected
5. Header on the canonical path
6. Epoch boundary
7. 2 more valid headers

The corpus pins:
- Expected event sequence (kinds, with key fields).
- Expected final `OrchestratorState.chain_dep.evolving_nonce`.
- Expected final `OrchestratorState.selector.current_tip_block_no`.

### Live interop crate

```rust

[package]

```

`src/bin/live_consensus_session.rs`:
- Connects to a cardano-node peer via N-A.
- Subscribes to chain-sync.
- Drives the orchestrator with the resulting `HeaderArrival` /
  `RollBack` events.
- For a configurable window (default: 1000 headers or 30 minutes),
  asserts:
  - Ade's `current_tip` block_no equals the peer's `Tip` from
    chain-sync at every chain-sync `MsgFound`.
- Emits a transcript log to stdout that the closure-gate test
  redirects to `docs/clusters/PHASE4-N-B/CE-N-B-6_<date>.log`.

`tests/live_consensus_session.rs`:
- One `#[test] #[ignore]` test:

## 14. Hard Prohibitions

### Inherited (from cluster.md)
- BLUE receiving `&ChainDb`, `&Mux`, parsing genesis text
- Wall-clock reads in BLUE
- `HashMap` / `HashSet`
- Float
- TODO/placeholder error variants
- `async fn` / `tokio` in BLUE

### Slice-specific
- `tokio` is permitted in `ade_core_interop` (RED) and `ade_runtime`
  *orchestrator* glue (GREEN); not in `ade_core::consensus::*`.
- `chain_selector.rs` is GREEN, so it may use `BTreeMap` if needed;
  must not use `HashMap`.
- No global mutable state in the orchestrator — `OrchestratorState`
  is the only mutable surface.
- The interop test must not run in CI by default (`#[ignore]`); CI
  runs the corpus driver only.

## 17. Review Notes

- The orchestrator's "single-header candidate fragment" approach
  works for the synthetic test but is a simplification — in a real
  network setting, fork-choice may receive multiple competing
  candidate fragments per slot. Closing CE-N-B-1 already exercised
  multi-candidate; this slice's CE-N-B-5 corpus deliberately stays
  linear-plus-one-rollback to keep the test surface small.
- The interop CE-N-B-6 is gated by `#[ignore]`. The Docker pin is
  inherited from S-A10's pattern; the log path is reserved.
