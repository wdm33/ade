## 1. Primary Invariant
**`DC-NODE-37`** (proposed) — when `select_best_chain` (`DC-CONS-03`) picks a winner that forks below Ade's current durable tip, **Ade never commits a rollback of its current durable chain until the replacement branch's bodies are fetched, linked, and validated as a complete candidate branch**; adoption then proceeds **only** via the already-enforced authorities (`materialize_rolled_back_state` `CN-STORE-07` + lockstep `receive::reducer` `DC-CONS-20` + `pump_block` `DC-NODE-05/12`), the fork anchor canonically bound to Ade's durable stored slot+hash (`DC-NODE-29`), recorded as `WalEntry::RollBack{ForkChoiceWin}`. Supporting chain: **`DC-NODE-34`** (peer identity preserved end-to-end), **`DC-NODE-35`** (candidate fragments derived only from `validate_and_apply_header`, never peer-minted; deterministic ordering; malformed/missing fails closed), **`DC-NODE-36`** (the `NeedsForkChoice` arm routes the candidate **set** to the single existing `select_best_chain` — no second selector, arrival-order-independent). **The BLUE fork-choice / validation / rollback core is reused, not built** — hypothesis **zero new BLUE canonical type** (`RollbackReason::ForkChoiceWin` already exists), *not zero-proof* (see §8/§10).

## 4. What Changes (design) — wiring, not building
**The gap:** `run_participant_sync` **fails closed** on `NeedsForkChoice` (`Err(UnexpectedRollback)`, `node_lifecycle.rs:2562-2569` — *"multi-candidate selection is a later multi-peer slice"*); the feed flattens peer identity (`NodeSyncItem = Block(Vec<u8>) | RollBack(Point)`, `node_sync.rs:77-80`); nothing live calls `select_best_chain`; the winner's bodies are never range-fetched as a selected branch.

- **S1 — peer-identity restoration (RED/GREEN, no BLUE).** Thread `peer` through `NodeSyncItem` + the `from_wire_pump`/`next_item` conversion (`node_sync.rs:203/:217` currently discard it). FOLLOW (single-peer) admission/replay **byte-unchanged**. **Non-goal:** S1 restores identity only — no selection/admission/rollback/evidence-verdict change. *(`DC-NODE-34`.)*
- **S2 — BLUE-safe per-peer candidate aggregation (color pending S2 proof; reuses BLUE `validate_and_apply_header`).** A per-peer aggregator groups peer-tagged events and assembles `CandidateFragment`s **only from `validate_and_apply_header` output** (the `chain_selector::process_header_arrival` validate-then-fragment pattern — NOT the `follow.rs` mint). Latent until S3. *(`DC-NODE-35`.)* **Entry gate (load-bearing — no live `select_best_chain` call may be introduced until candidate construction proves):** (1) peer identity preserved; (2) **hash-critical protocol paths use preserved original wire bytes, internal candidate comparison/proof surfaces use project-canonical bytes**; (3) fragments derived from Ade validation, not peer claims; (4) candidate ordering deterministic; (5) malformed/missing fails closed. *Keeps the cluster from becoming "look over the peer's shoulder."*
- **S3 — live selector dispatch (RED/GREEN + BLUE-reused).** Wire the `NeedsForkChoice` arm to call `select_best_chain` over the aggregated candidate **set**; `TiebreakerLossKeepCurrent` → no durable change; a `ChainSelected` → S4 apply. No second selector; live arrival-order-independent. *(`DC-NODE-36`.)*
- **S4 — selected-range fetch + fork-switch apply (RED fetch + GREEN sequencing + BLUE-reused; the security-critical slice).** Block-fetch the winner's bodies (`RequestRange` anchor→tip from the winning peer; fork anchor bound to the durable stored slot+hash, `DC-NODE-29`), **validate them as a complete linked branch first**, then commit `RolledBack(fork_anchor)+ChainSelected(body)×N` via `apply_chain_event` (`ForkChoiceWin`). **Hard line: the current durable chain is never abandoned until the replacement branch is fetched, linked, and validated.** A failed/lying/incomplete winner leaves the current chain unchanged. *(`DC-NODE-37`.)*
- **S5 — reselection replay-equivalence + reconcile + forge-fence (BLUE-reused + GREEN/RED).** A `ForkChoiceWin` reselection replays byte-identically (multi-peer corpus); `selector.current_tip == ChainDb::tip` after every decision; no forge across pending reselection. *(Strengthens `DC-NODE-26/27/28`.)*
- **S6 (operator pass) — CE-AO-6.** Two competing producers on the CE-AI-6 venue; live convergence captured + committed → flips `CN-CONS-03`. Operator-gated.

## 8. Forbidden During This Cluster (slice-level hard prohibitions inherit) — the nine hard lines
1. **No live `select_best_chain` dispatch (S3+) before S1 (peer identity) + S2 (BLUE-safe construction) are proven.** S2's 5-point entry gate is the hard gate.
2. **`select_best_chain` stays the SINGLE selector** — routed-to, never duplicated; no second selector / parallel preference / density ordering / operator heuristic.
3. **No RED-minted `ValidatedHeaderSummary` (the `follow.rs` shape) may reach `select_best_chain`** — candidate summaries only from `validate_and_apply_header`.
4. **`pump_block` stays the sole roll-forward durable admit** — no header-only tip advance; a win is provisional until bodies apply.
5. **NEVER commit a rollback of the current durable chain until the replacement branch's bodies are fetched, linked, and validated as a complete candidate branch** (`DC-NODE-37`; the H-1 class at fork-choice scale).
6. **The fork-anchor rollback target binds Ade's durable stored slot+hash** (`DC-NODE-29`) — never peer-supplied, never mixed peer/local authority.
7. **The `ForkChoiceWin` reselection is durably WAL-recorded + replay-equivalent** — reuse `materialize`/`commit_rollback`/the WAL marker; NOT a second rollback implementation.
8. **Venue stays explicit + closed** (Participant only reaches SELECT; `SingleProducer`/`Unknown` fail closed); **`CN-CONS-03` flip is operator-gated CE-AO-6** — proves the exercised two-producer venue, NOT full Cardano ChainSel.
9. **Hypothesis ≠ license:** the aggregator's color is UNRESOLVED until S2 proves it; **no new BLUE canonical type** without explicit re-scoping; **no `RO-LIVE` flip** beyond the operator-gated CE-AO-6.
