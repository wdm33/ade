# Invariant Slice — B3-S4

## 1. What an Invariant Slice Is
A mergeable unit that strengthens exactly one invariant family and leaves the system fully
correct and replay-equivalent on merge. This slice removes the conservation early-out and wires
the full equation; it is where the known false-accept path dies.

## 9. Design Summary

The full coin-level equation, all terms `i128`, no float:

```text
consumed = Σ(resolved input coins) + Σ(withdrawals) + refunded_deposits
produced = Σ(output coins) + fee + donation + new_deposits
accept ⟺ consumed == produced
```

- `Σ(withdrawals)` from B3-S3 `withdrawals_sum`.
- `new_deposits` / `refunded_deposits` from B3-S2 `classify` over `decode_conway_certs`, summing
  the `DepositEffect::NewDeposit`/`Refund` coins (sourced per `CoinSource`: explicit-in-cert,
  `ConwayDepositParams`, or registration state).
- `donation` from `body.donation` (a produced term, per sketch D5); `treasury_value` carries no
  conservation weight (D5 — reference-confirmation item, asserted = no weight here).

### 9.1 Reject precedence (BINDING)

When a tx triggers more than one failure, the composer rejects with the **highest-precedence**
reason, deterministically and independent of evaluation order. The pipeline order *is* the
precedence (so determinism is by construction):

1. **Decode failure** — `CodecError` (body / certs (B3-S2) / withdrawals (B3-S3)).
2. **Era-invalid cert** — `CertDisposition::NotValidInConway` (B3-S2).
3. **Missing validation environment** — `ValidationEnvironmentError::MissingConwayDepositParams`
   (B3-S1).
4. **Unsupported state-dependent accounting** — `UnsupportedStateDependentDepositAccounting`
   (B3-S2/S4).
5. **Value not conserved** — `ConservationError` (this slice).

> Pre-existing composer checks (collateral, network id, required signers) retain their current
> placement; §9.1 fixes the precedence **among the accounting-related rejects** and pins
> conservation as the final value check. The composer must not reorder the accounting pipeline
> such that a lower-precedence reason can mask a higher one.

## 13. Failure Modes
Per §9.1, in precedence order — all fail-fast, deterministic, structured:
1. `CodecError` (decode).
2. `CertDisposition::NotValidInConway` → structured Invalid (era-validity).
3. `ValidationEnvironmentError::MissingConwayDepositParams` (environment, not tx-validity).
4. `UnsupportedStateDependentDepositAccounting` (accounting-environment insufficiency).
5. `ConservationError { consumed_coin, produced_coin }` (value not conserved).

A tx triggering several rejects with the highest-precedence reason; replay-stable.

## 14. Hard Prohibitions

### Inherited Cluster-Level Prohibitions
All "Forbidden During This Cluster" items from `docs/clusters/PHASE4-B3/cluster.md` — notably:
re-introducing any early-out / silent skip on cert/withdrawal txs; guessing a state-dependent
deposit instead of structured-reject; positive-only coverage; `HashMap`/float/wall-clock in BLUE;
**a false accept is release-blocking**.

### Slice-Specific Prohibitions
- Leaving (or re-adding) any early-out that bypasses the equation for cert/withdrawal-bearing txs.
- Reject precedence that depends on evaluation order (must match §9.1 deterministically).
- Letting a lower-precedence reason mask a higher one (e.g. reporting `ConservationError` when the
  certs failed to decode).
- Softening a B2 corpus assertion to make the equation pass (CE-B2-3/4 must stay green honestly).
- Counting `treasury_value` as a conservation term, or omitting `donation` from produced.
- Floating point / `HashMap` in the accounting path.

**If any of these appear, the slice is incorrect.**
