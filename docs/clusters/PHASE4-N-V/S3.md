## OUTCOME (DECIDED): OQ4 honest fallback — CE-V-6 deferred to follow-on N-W

Driving to `ForgeSucceeded` uncovered a **second** latent producer/validator
asymmetry behind the envelope: the producer builds the leader VRF proof over the
**TPraos** role-tagged input `vrf_input(slot, eta0, LeaderEligibility)`
(`slot‖eta0‖0x4C`), but **Conway is Praos** — `validate_and_apply_header` →
`verify_praos_vrf` verifies over `praos_vrf_input(slot, eta0)`
(`blake2b256(slot‖eta0)` + `vrfLeaderValue` range-extension). The alphas differ,
so a consistent eligible-leader tick reaches header-validation step 6 and is
rejected with `BlockValidityError::Header(VrfCert(VerificationFailed))`. The
keyhash binding (step 5) passes — every test-controlled binding is correct — so
this is a genuine BLUE producer-side bug, not a setup issue.

**Per the OQ4 hard rule and user re-scope decision: N-V does NOT fake
`ForgeSucceeded`.** N-V closes on the envelope fix (S1+S2). CE-V-6 + the
producer-side TPraos→Praos VRF migration (~4 authorities: `run_real_forge`,
`verify_and_evaluate_leader`, `query_leader_schedule`, the
`LeaderScheduleAnswer.expected_vrf_input` contract) become a **new follow-on
cluster (N-W)**. The honest-fallback test below pins the blocker; it must be
promoted to assert `ForgeSucceeded` when N-W lands.

**Test shipped:** `crates/ade_node/tests/forge_succeeds.rs` →

## Goal

An integration test `forge_to_self_accept_succeeds` (in `crates/ade_node/tests/`)
that drives a **consistent eligible-leader** tick through
`produce_mode::run_real_forge` and asserts `CoordinatorEvent::ForgeSucceeded`.
This is the first time a forged block passes `self_accept` in-process.

## Why it's reachable now (and was not in N-T)
N-T's S5 Tier-A attempt built this setup but failed at the **envelope** decode
(`run_real_forge` step at produce_mode.rs:749 → `decode_block` →
`Body(InvalidStructure)`), before `self_accept`. PHASE4-N-V S2 wrapped
`forge_block` output in `[era, block]`, so the decode now passes and the tick
reaches `self_accept`'s header validation. The N-S-A KES-signs-unsigned-header
bridge already lands the real KES sig.

## Consistent eligible-leader recipe (mirror `forge_handler_variants::synth_shell`)
The N-T full-stake test failed self_accept only because its `pool_distr_view`
was **empty** (unknown pool → VRF-cert reject). S3 registers the operator pool:
- `shell = synth_shell(..)`; `vrf_vk = shell.vrf_verification_key()`;
  `cold_vk = shell.cold_vk()`.
- `pool_id = Hash28(ade_crypto::blake2b_224(&cold_vk.0).0)`.
- `vrf_keyhash = Hash32(ade_crypto::blake2b_256(&vrf_vk.0).0)` — **confirm this
  is the recipe `block_validity` header validation binds**: it checks
  `Hash32(blake2b_256(header.vrf_vk)) == pool_vrf_keyhash` (header_validate.rs;
  verify the exact call before relying on it — OQ4).
- `pool_distr_view = PoolDistrView::new(epoch, total, ActiveSlotsCoeff{1,1}, { pool_id: PoolEntry { active_stake: total, vrf_keyhash } })`, `total > 0` (e.g. 1) so ASC 1/1 ⇒ always eligible regardless of VRF output bytes.
- `leader_answer = LeaderScheduleAnswer { slot, pool: pool_id, epoch, expected_vrf_input: vrf_input(slot, eta0, VrfRole::LeaderEligibility), stake_fraction: (1,1), asc: {1,1} }` (or derive via `query_leader_schedule(&{slot,pool_id}, &pool_distr_view, &era_schedule, &chain_dep)`).
- `eta0` = a fixed `Nonce`; `chain_dep = PraosChainDepState::empty()` with `epoch_nonce = eta0`; `ctx.eta0 = &chain_dep.epoch_nonce` (same nonce both places).
- `era_schedule`: single Conway `EraSummary`, `start_epoch = epoch`, `start_slot ≤ slot`. `base_state = LedgerState::new(Conway)` with `epoch_state.epoch = epoch`.
- `ctx = ForgeRequestContext { eta0, vrf_vk, leader_schedule_answer, pparams: default, base_state, chain_dep_state, era_schedule, pool_distr_view, block_number: BlockNo(0), prev_hash: Hash32([0;32]), protocol_version {9,0}, prev_opcert_counter: None }`.
- `let event = run_real_forge(slot, /*kes_period*/ 0, &ctx, &mut shell);`
- **Assert `matches!(event, CoordinatorEvent::ForgeSucceeded { .. })`.**

## Diagnostic path (if it does NOT succeed first try)
`run_real_forge` maps every failure to `ForgeFailureReason::Other`/`SelfAcceptRejected`, hiding the cause. To see the exact `BlockValidityError`:
- Temporarily replicate the call's tail (or add a throwaway `eprintln!` in `run_real_forge` behind a test) to surface the `self_accept(...)` `Err(SelfAcceptError::Rejected(BlockValidityError))`. Read the typed error.
- Most likely mismatches + their fixes (all **test-setup** issues, not BLUE):
  - VRF-cert reject → `vrf_keyhash` recipe wrong, or `pool_id` derivation mismatched between answer/view/header-issuer. Align to what header validation actually computes.
  - VRF proof verify fail → `eta0`/`slot` used in `vrf_input` differs between forge and validation. Use `chain_dep.epoch_nonce` consistently.
  - KES/opcert reject → opcert period window vs `kes_period`; use `synth_shell`'s opcert at period 0 + `kes_period = 0`.
  - epoch mismatch → `era_schedule.start_epoch` ≠ `pool_distr_view` epoch ≠ `base_state.epoch_state.epoch`.
- **Revert any throwaway instrumentation before finishing.**

## OQ4 honest fallback (HARD RULE)
If, after aligning the setup, the block still cannot `self_accept` because of a
**genuine BLUE validation bug** (not a test-setup issue) — i.e., fixing it needs
real header/forge-authority surgery beyond an in-slice correction — **STOP. Do
NOT fake `ForgeSucceeded`. Do NOT loosen the assertion.** Root-cause the exact
`BlockValidityError`, write it up, and report it for re-scope. A correct
"`ForgeFailed` with the precise next blocker documented" is an acceptable
honest outcome that halts the cluster's CE-V-6; faking success is not.
