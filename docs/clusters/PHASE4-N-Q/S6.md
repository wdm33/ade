# Slice S6 — Operator runbook + registry close

**Cluster:** PHASE4-N-Q
**Strengthens:** `CN-CONS-06`, `RO-LIVE-01` (both
strengthened_in += "PHASE4-N-Q" — the producer-mode
composition surface is in place; the bounty-facing live half
remains operator-action gated on the follow-on forge cluster).
**Flips to enforced:** `CN-PROD-01`, `CN-PROD-02`, `DC-PROD-01`,
`DC-PROD-02` (all four declared in S1, mechanical claims now
backed by S2..S5 tests).
**Introduces:** no new code; this slice ships docs + registry
flips + cluster close.
**Dependencies:** PHASE4-N-Q S5 (HEAD `7ee2723`).

## §1 Intent

Close PHASE4-N-Q honestly: document what the cluster ships (the
producer-mode *composition surface* — listener + slot loop +
GREEN coordinator + RED key-custody shell + closed evidence
vocabulary) and document what's still operator-action work
(real forge handler + ServedChainSnapshot push + cardano-cli
opcert envelope parser + full Conway genesis parser + the
actual cardano-node-accepts-Ade-block evidence capture).

The bounty artifact is *not* in S6's scope. The cluster's
title is "live producer-mode" and what it delivers is the
real engineering surface the operator needs — not the
operator pass itself. This split mirrors PHASE4-N-G's
precedent (mechanical adapter shipped + operator-action
log deferred via `RO-LIVE-01.open_obligation`).

## §3 Doctrine alignment

- [[feedback-hard-closure-gates]] — the four N-Q rules
  are at hard gates (enforced) backed by tests + (where
  available) CI scripts. The carry-forward into the
  follow-on forge cluster is recorded as `open_obligation`,
  not as a partial status.
- [[feedback-bounded-smoke-slices]] — the S5 smoke is bounded
  (3 slots, synthetic keys, no peer). S6's evidence-capture
  template documents the bounty-facing operator pass as
  *additional* operator-action work, not a substitute for
  the bounded smoke.
- [[feedback-shell-must-not-overstate-semantic-truth]] —
  the registry's open_obligation fields are explicit about
  the gap between "listener handshakes" (S4 fact) and "Ade
  block accepted by cardano-node" (follow-on forge cluster).
  No claim in the registry overstates what the producer-mode
  composition surface achieves on its own.
- [[project-bounty-requirements]] — the bounty artifact
  (cardano-node accepts Ade-forged block) is honestly named
  as out-of-scope-for-N-Q in both the runbook and the
  CN-CONS-06 / RO-LIVE-01 open_obligations.

## §4 Exit criteria

1. `docs/active/cn-cons-06-operator-runbook.md` exists and
   is reproducible from cluster-local context.
2. `docs/clusters/PHASE4-N-Q/CE-N-Q-OPERATOR_PROCEDURE.md`
   exists.
3. Registry: CN-PROD-01/02 + DC-PROD-01/02 status =
   `enforced`; CN-CONS-06 + RO-LIVE-01 strengthened_in
   includes `"PHASE4-N-Q"`.
4. Every open_obligation names a concrete reopen criterion.
5. `cargo test --workspace --lib` clean (no regressions).
6. `ci/ci_check_producer_coordinator_no_secrets.sh` passes.
7. `git push origin main` succeeds.
