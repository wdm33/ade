# Slice A4 — Replay-equivalence harness + CI

**Cluster:** PHASE4-N-M-A
**CEs:** CE-N-M-A-6 (DC-WAL-03)
**Dependencies:** A1, A2, A3.
**Registry effects on merge:** DC-WAL-03 → `enforced`.

## Intent

Prove the headline runtime contract:

> `same BootstrapAnchor + same block-bytes inputs + same WAL →
> byte-identical final LedgerState fingerprint across two
> replays.`

## Scope

- `crates/ade_runtime/tests/wal_replay_from_anchor.rs` —
  integration test.
- Test harness:
  - Synthesize a small `BootstrapAnchor` (cold-start-style
    genesis + 3-entry mini UTxO seed).
  - Build N `WalEntry::AdmitBlock` entries from the Conway
    corpus blocks (pick 3 lightest blocks via
    `pick_lightest_blocks(corpus, 3)`).
  - Two runs:
    - Run A: replay anchor + WAL through
      `replay_from_anchor`, capture final ledger fingerprint.
    - Run B: same.
  - Assert byte-identical.

## §12 Mechanical Acceptance Criteria

- [ ] `wal_replay_from_anchor_two_runs_byte_identical` — two
  runs over the same `(anchor, WAL)` produce equal final
  ledger fingerprints.
- [ ] `wal_replay_from_anchor_post_fp_matches_wal_tail` — final
  ledger fingerprint equals `WAL[N].post_fp`.
- [ ] `wal_replay_from_anchor_rejects_chain_break` — corrupt one
  `prior_fp`; replay returns `WalReplayError::ChainBreak`.
- [ ] `wal_replay_from_anchor_rejects_missing_block_bytes` —
  WAL has entry for block-hash H but block_bytes map lacks H;
  → `WalReplayError::BlockBytesMissing`.

## §14 Hard Prohibitions

- No use of wall-clock / rand in the harness.
- No skipping `verify_chain` before replay.
- No HashMap.

## §15 Non-Goals

- No live block-fetch in the test (we use the Conway corpus
  block bytes, which are already in the test fixtures).
- No multi-WAL-file replay (one file is enough for 3 entries).
