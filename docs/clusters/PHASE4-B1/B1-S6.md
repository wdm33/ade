## 9. Design Summary

For each of the 14 corpus blocks:
1. Build inputs: `PoolDistrView` (from corpus, B1-S2 helper), a
   `PraosChainDepState` seeded with `epoch_nonce = eta0(576)` and permissive
   monotonic fields (`last_slot = None` so each block validates independently),
   the mainnet `EraSchedule`, and a Conway `LedgerState` with `track_utxo=false`
   (structural body validation; same setup as B1-S4's `valid_block_evolves_both_states`).
2. `block_validity(...)` → expect `Valid`.
3. Record `encode_verdict_surface(&verdict)`.

**Replay equivalence**: run the whole sequence twice; assert the two vectors of
verdict-surface bytes are byte-identical.

### Plutus-eval risk (report honestly)

The 14 blocks are real mainnet Conway blocks and may contain Plutus txs whose
**full** evaluation hits the known, externally-blocked CE-88 / aiken limitation
(`project_aiken_main_bug_unfixed`). If `apply_block_with_verdicts` returns a
body error ONLY because of that known limitation:
- Do NOT fake a Valid and do NOT weaken `block_validity`.
- Report exactly which blocks fail and why.
- CE-B1-3 then closes for **header + body-structural + body-hash** validity over
  all 14, with full Plutus-eval agreement explicitly carrying the pre-existing
  CE-88 limitation (documented in the test + corpus README) — NOT a B1 regression.
If all 14 are fully `Valid`, CE-B1-3 closes outright.

The replay-equivalence assertion (CE-B1-5) holds regardless of Valid/Invalid —
it asserts the verdict stream is identical across runs.
