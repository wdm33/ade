# Slice S5 — `KesSecret.inner` migration + loader acceptance + cluster close

**Cluster:** PHASE4-N-P
**Strengthens:** I7 (loader determinism), N8 (no `Ok(_)` on
malformed), N9 (no upstream-shim in production), R3 (cardano-cli
envelope and Ade-native envelope are observationally equivalent
post-load). Closes OP-OPS-04's cardano-cli-import obligation +
DC-CRYPTO-07's PHASE4-N-P deferral.
**Flips to `enforced`:** DC-CRYPTO-08, DC-CRYPTO-09.
**Strengthens:** DC-CRYPTO-04, DC-CRYPTO-05.
**Dependencies:** S1 (proof obligation), S2 (BLUE algorithm), S3
(serde), S4 (cardano-cli corpus + compatibility CI gate).

## §1 Intent

Migrate `KesSecret.inner` from `cardano_crypto::kes::Sum6Kes::SigningKey`
to `ade_crypto::kes_sum::Sum6Kes::SigningKey`, drop the `kes-sum`
feature from `ade_runtime`'s `cardano-crypto` declaration, and flip
`load_kes_signing_key_skey` from "fail-closed always" to
"608-byte structurally-valid → `Ok(KesSecret)`; everything else
→ fail-closed via closed `KeyLoadError` variants".

After S5:
- No `cardano_crypto::kes::*` imports remain in `ade_runtime`
  production code.
- `ade_runtime/Cargo.toml`'s `cardano-crypto` features list does
  not contain `"kes-sum"`.
- The cardano-cli expanded-import path is fully supported (and
  mechanically validated against the S4 ground-truth corpus).
- `OP-OPS-04.open_obligation = null` (cleared).
- `DC-CRYPTO-07.open_obligation = null` (cleared).

## §2 Scope

### Migration

- `crates/ade_runtime/src/producer/signing.rs`:
  - `use cardano_crypto::kes::{KesAlgorithm, Sum6Kes}` →
    `use ade_crypto::kes_sum::{KesAlgorithm, KesError, Sum6Kes}`.
  - `KesSecret.inner: <Sum6Kes as KesAlgorithm>::SigningKey` now
    resolves to the BLUE-owned `SumSigningKey<Sum5Kes>`.
  - `SigningError::CardanoCrypto(...)` retained for the VRF path
    (out of N-P scope); new `SigningError::AdeKesSum(KesError)`
    for the BLUE KES path.
  - `gen_key_kes_from_seed_bytes`, `sign_kes`, `update_kes`,
    `derive_verification_key` all switch to the BLUE API
    (notably: `sign_kes(&sk, period, msg)` vs upstream's
    `sign_kes(&(), period, msg, &sk)`; `update_kes` takes period
    by value, not `&()` context; `derive_verification_key`
    returns `[u8; 32]` directly, not a `Result<Vec<u8>>`).
  - New `pub(super) fn KesSecret::from_blue_signing_key(inner,
    period)` constructor used by the loader.

- `crates/ade_testkit/src/producer/reference_vectors.rs`:
  - Same migration. KES reference vectors are now materialized via
    the BLUE algorithm; they CHANGE byte-for-byte vs N-C-era
    values because our `expand_seed` prefix bytes differ from
    `cardano-crypto` Rust 1.0.8's (Haskell-correct 0x01/0x02 vs
    upstream's 0x00/0x01). Internally consistent: signing.rs
    tests still pass.

- `crates/ade_runtime/src/producer/keys.rs`:
  - `load_kes_signing_key_skey(path)` body replaced with the new
    accept-608-valid policy:
    - `read_envelope_payload(KES_SIGNING_KEY_TYPE)`
    - `if payload.len() != 608 → UnsupportedExpandedKesKeyFormat`
    - `Sum6Kes::raw_deserialize_signing_key_kes(&payload)`
      (returns the BLUE structural validation; closed errors map
      to `KeyLoadError::KesParse(KesParseError::*)`)
    - `current_period_of_signing_key(&inner)` (S3 helper)
    - `KesSecret::from_blue_signing_key(inner, period)`
  - New `KeyLoadError::KesParse(ade_crypto::kes_sum::KesParseError)`
    variant.
  - Test updates: `cardano_cli_kes_envelope_rejects_608_byte_payload`
    renamed to `cardano_cli_kes_envelope_rejects_synthetic_608_byte_payload`
    (synthetic [0xAB; 608] now fails via `KesParse`, not the size
    variant); new tests `cardano_cli_kes_envelope_accepts_real_608_byte_payload`
    and `cardano_cli_kes_envelope_rejects_608_byte_leaf_zero_payload`.

- `crates/ade_runtime/Cargo.toml`:
  - `cardano-crypto = { ..., features = ["vrf-draft03", "dsign"] }`
    (drop `"kes-sum"`).

### CI gate updates

- `ci/ci_check_kes_envelope_closed.sh` Guard 2:
  - Now requires `raw_deserialize_signing_key_kes` to appear in
    `load_kes_signing_key_skey`'s body (the accept path).
  - Still forbids `KesSecret::from_bytes_zeroizing` /
    `KesSecret::from_seed_at_period` in the loader body (would
    bypass the BLUE structural validator).
  - Still requires `UnsupportedExpandedKesKeyFormat` for the
    size-mismatch branch.

## §4 Hard prohibitions

- **N8.** No `Ok(_)` from `load_kes_signing_key_skey` on a payload
  that isn't a structurally-valid 608-byte expanded `Sum6KES`.
  Enforced by `ci_check_kes_envelope_closed.sh` Guard 2.
- **N9.** No `cardano_crypto::kes` imports in `ade_runtime/src/**`
  production code; no `kes-sum` feature on `ade_runtime`'s
  `cardano-crypto`. Enforced by `ci_check_kes_sum_compatibility.sh`
  (KES-scoped — VRF and DSIGN paths remain on upstream until a
  future cluster).
