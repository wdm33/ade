# Invariant Slice — B5-S5

## Slice Header

**Slice Name:** Hardening — checked DRep-expiry arithmetic (deterministic fail-closed on overflow)

**Cluster:** PHASE4-B5 — Conway governance certificate accumulation authority

**Status:** Merged

**Origin:** Per-cluster security review (cluster-close, step 5) raised a BLOCK on
the unchecked `current_epoch + drep_activity` in `gov_cert.rs`. Because the

**Cluster Exit Criteria Addressed:**
- [ ] **CE-3 / CE-6 (hardening):** DRep expiry uses `checked_add`; an overflowing
  `drep_activity` is a deterministic `DRepActivityOverflow` fail-fast, never a
  silent wrap. Defended by a unit test, a corpus adversarial test, and a CI guard
  forbidding bare `+` on the expiry.

---

## Rationale

The workspace has no `overflow-checks` profile, so a release build would wrap a
`current_epoch + drep_activity` overflow silently to a wrong (small) expiry — a
debug-vs-release determinism divergence and a silent gov-state corruption rather
than a deterministic halt. The rest of the ledger arithmetic surface uses
`checked_add`/`saturating_add`; this restores that convention on the new
authoritative path.

`drep_activity` originates from a protocol parameter, so an overflow indicates an
ill-formed *environment* (not a bad transaction) — hence the new
`ValidationEnvironmentError::DRepActivityOverflow`, a sibling of
`MissingDRepActivityParam`.
