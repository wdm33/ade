## Invariant scope

CN-FORGE-04 is mechanically protected: a CI gate rejects any reintroduction of
the TPraos leader alpha on the producer path, any both-alphas fallback, removal
of TPraos validation, or loss of the TPraos-producer fail-closed; and the
registry records CN-FORGE-04 as `enforced` with its tests + CI gate bound.

## Changes

### CI gate — `ci/ci_check_producer_praos_vrf.sh` (new)
Six guards (all pass at HEAD):
- **G1** `leader_vrf_input` (single era→construction authority) defined exactly
  once, in `vrf_cert`.
- **G2** no bare `vrf_input(` on the producer leader path (produce_mode /
  leader_schedule / leader_check) — they route through `leader_vrf_input` /
  the answer's `alpha_bytes`.
- **G3** TPraos construction preserved: `vrf_input` + `VrfRole` still in
  `vrf_cert` (validation not removed — CE-W-7).
- **G4** TPraos producer forging fail-closes: `UnsupportedProducerEra` present
  (producer_log) + used (produce_mode) — CE-W-6.
- **G5** no both-alphas fallback: no file outside `vrf_cert.rs` contains both
  `praos_vrf_input(` and a bare `vrf_input(` (the dual construction lives only
  inside `leader_vrf_input`) — N1.
- **G6** Praos eligibility threshold uses `leader_value_for` (era-correct
  leader value), not the raw VRF output — CE-W-2.
