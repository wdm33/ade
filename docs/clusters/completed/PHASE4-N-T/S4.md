## Goal

Stop dropping the `AcceptedBlock` token; route it to `ServedChainHandle::push_atomic`
so forged blocks become servable. Fail-closed via `BroadcastPushError`. Emit the
existing closed `ProducerLogEvent::BlockServed` for blocks observed-present in the
served snapshot on a block-fetch.

## Edits (all `crates/ade_node/src/produce_mode.rs`)

### 1. New closed error
```rust
#[derive(Debug, Clone, PartialEq)]
pub enum BroadcastPushError {
    /// The self_accept replay (inside ChainEvolution::advance) rejected
    /// the forged bytes — push_atomic is NOT called.
    SelfAcceptReplayRejected,
    /// ServedChainHandle::push_atomic itself failed.
    Push(ade_runtime::producer::served_chain_handle::PushError),
}
```

### 2. push_atomic wiring (CE-T-10)
- Un-discard the handle at `:284`: `let (served_chain_handle, served_chain_view) = ...;`. Thread `&served_chain_handle` into `apply_effects_with_forge_handler` (new param) at its call site(s).
- In `apply_effects_with_forge_handler`'s `RequestForge` arm: declare `let mut pending_accepted: Option<AcceptedBlock> = None;` local to the arm. In the advance `Ok((next_evo, accepted))` branch (`:1016`) set `pending_accepted = Some(accepted);`. In the advance `Err(_)` branch (`:1020`), construct `BroadcastPushError::SelfAcceptReplayRejected`, log it (`eprintln!`), and keep feeding `ForgeFailed` (no broadcast — push_atomic never called).
- In the `more_effects` loop (`:1038`), add a `CoordinatorEffect::BroadcastBlock { .. }` arm: `if let Some(accepted) = pending_accepted.take() { match served_chain_handle.push_atomic(accepted) { Ok(_tip) => {} , Err(e) => { eprintln!("... {:?}", BroadcastPushError::Push(e)); } } }`. (BroadcastBlock with no pending token — unreachable — is a defensive no-op.)

### 3. BlockServed via served-snapshot observation (A5 / CN-PROD-04)
In `dispatch_server_frame_event_to_outbound`'s `PeerN2nServerBlockFetchFrame` arm, emit `BlockServed` for the blocks the request covers that are **present** in the served snapshot — observed independently of the opaque `ServerReply`:
- Decode the `BlockFetchMessage` request range (the `from`/`to` points carry `slot`+`hash`).
- Borrow `served_chain_view`; for each block in the requested range present in the snapshot, collect `(slot, hash, bytes_len)` into a `Vec` **before** any `.await` (do NOT hold the `watch::Ref` across `.await`).
- Emit one `ProducerLogEvent::BlockServed { peer_id, slot, hash, bytes_len }` per collected block.
- **Evidence access:** `dispatch_server_frame_event_to_outbound` does not currently take the evidence writer. Either (a) thread `evidence_writer: &mut std::fs::File` into it, or (b) have it return the collected `Vec<BlockServed-tuple>` and let `handle_listener_event` (which has `evidence_writer`) emit them after the call. **Prefer (b)** — it avoids widening the async fn's mutable-state surface and keeps the `watch::Ref`-across-`await` hazard contained. Extend the return type from `Result<usize, DispatchError>` to also carry the served-block evidence (e.g., `Result<(usize, Vec<ServedBlockEvidence>), DispatchError>` with a small local `struct ServedBlockEvidence { peer_id, slot, hash, bytes_len }`), and emit in `handle_listener_event`.
- **Honest-scope:** this is GREEN evidence *observation* of the served snapshot (only present blocks), separate from the BLUE serve authority (`producer_block_fetch_serve`). Per `feedback_evidence_reducers_are_green_not_authority`, evidence observes; it does not re-decide what is served. If precisely mirroring the reducer's partial-overlap subset proves infeasible without a `ServerReply` accessor, emit `BlockServed` only for blocks whose `(slot,hash)` key is present in the snapshot AND within the requested range — never over-claim.
