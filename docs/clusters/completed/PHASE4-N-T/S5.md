## Goal

A hermetic loopback integration test proving N-T's end-to-end wiring:
**a self-acceptable block, threaded through the real `ChainEvolution::advance`
path and pushed via `ServedChainHandle::push_atomic`, is present in the served
snapshot and reads back byte-identically through the served-chain lookup** (the
same lookup the block-fetch serve path uses), with two-run replay determinism.

## Two-tier approach (attempt the strong one; fall back honestly)

**Tier A (preferred — full produce forge).** Construct a consistent eligible-leader
setup and drive `produce_mode::run_real_forge` to `ForgeSucceeded`, then feed that
artifact through `ChainEvolution::advance` → `push_atomic` → served readback.
Recipe (mirror `crates/ade_node/tests/forge_handler_variants.rs::synth_shell`):
- `shell = synth_shell(..)`; `vrf_vk = shell.vrf_verification_key()`;
  `pool_id = Hash28(ade_crypto::blake2b_224(&shell.cold_vk().0).0)`;
  `vrf_keyhash = ade_crypto::blake2b_256(&vrf_vk.0)` (confirm the keyhash recipe
  the VRF-cert validator uses — `ade_crypto::blake2b` + `ade_core::consensus::vrf_cert`).
- `pool_distr_view = PoolDistrView::new(epoch, total, ActiveSlotsCoeff{1,1}, { pool_id: PoolEntry{ active_stake: total, vrf_keyhash } })` with `total` > 0 and ASC 1/1 so the leader threshold is 1 (always eligible regardless of VRF output bytes — see `kes_period_outside_window_emits_forge_failed_kes_period_mismatch` which used (1,2)+(1,1) to reach Eligible).
- `era_schedule` single Conway entry with `start_epoch = epoch`, `start_slot ≤ slot`; `base_state = LedgerState::new(Conway)` with `epoch_state.epoch = epoch`; `chain_dep` with `epoch_nonce = eta0`; `eta0` arbitrary fixed nonce.
- `answer = query_leader_schedule(&LeaderScheduleQuery{slot, pool: pool_id}, &pool_distr_view, &era_schedule, &chain_dep)?`.
- `ctx = ForgeRequestContext { eta0: &chain_dep.epoch_nonce, vrf_vk: &vrf_vk, leader_schedule_answer: &answer, pparams: &ProtocolParameters::default(), base_state: &base_state, chain_dep_state: &chain_dep, era_schedule: &era_schedule, pool_distr_view: &pool_distr_view, block_number: BlockNo(0), prev_hash: Hash32([0;32]), protocol_version: {9,0}, prev_opcert_counter: None }`.
- `let event = run_real_forge(slot, 0, &ctx, &mut shell);`
- **If `event` is `ForgeSucceeded { artifact, .. }`:** seed a `ChainEvolution` from
  `(base_state, chain_dep, None, era_schedule, pool_distr_view, eta0)`,
  `let (evo2, accepted) = evo.advance(&artifact.bytes).expect("self-accepts");`
  then `push_atomic` + readback (below). Assert `event` was `ForgeSucceeded`.
- **If `event` is NOT `ForgeSucceeded`** (the eligible in-process forge does not
  self-accept — N-R-A established this was unproven; some header/VRF/body
  consistency requirement is unmet): **do NOT fail the test**; fall through to
  Tier B for the serve proof and record (test comment + report) that the
  in-process `ForgeSucceeded` demonstration remains the honest residual.

**Tier B (fallback — guaranteed serve proof via a real self-acceptable block).**
Obtain an `AcceptedBlock` exactly as `served_chain_handle.rs`/`broadcast_to_served.rs`
tests do: `self_accept(corpus_lightest_block, &ledger_at_576, &chain_dep(eta0=corpus.epoch_nonce), &schedule, &pool_distr_view_from_corpus)`. Seed a
`ChainEvolution` from that same corpus state and `advance(&corpus_block)` to get
the token through the **real advance path** (this is exactly what produce_mode does
on a `ForgeSucceeded`). A corpus block is a real, validator-accepted block —
equivalent to a forged one for the serve+fetch path.

## Served readback (both tiers)
```rust
let (handle, view) = ServedChainHandle::new();
let tip = handle.push_atomic(accepted).expect("push");
let snap = view.borrow();
let got = snap.block_bytes(tip.slot, &tip.hash).expect("present");
assert_eq!(got, &block_bytes[..], "served block reads back byte-identically");
```
(`ServedChainSnapshot::block_bytes(slot, &hash) -> Option<&[u8]>` is the serve-path
lookup — see `self_accept.rs` tests `served_chain_admit_admits_corpus_block`.)

## Two-run replay (CE-T-11 / DC-PROD-03)
Run the seed→advance→push sequence twice into two fresh `ServedChainHandle`s; assert
`view_a.borrow().fingerprint() == view_b.borrow().fingerprint()`.
