## §1 Intent

Land the RED shared-handle wrapper around the pure-value
`ServedChainSnapshot` that N-R-B B3/B4 wire into
`produce_mode`. The handle uses
`tokio::sync::watch::channel<ServedChainSnapshot>` per DQ-B1;
`push_atomic` returns a closed `ServedTip { slot, hash }` per
DQ-B2.

## §2 New surface

```rust
pub struct ServedChainHandle {
    sender: watch::Sender<ServedChainSnapshot>,
}

#[derive(Clone)]
pub struct ServedChainView {
    receiver: watch::Receiver<ServedChainSnapshot>,
}

#[derive(Debug, Clone, PartialEq, Eq, Hash)]
pub struct ServedTip {
    pub slot: SlotNo,
    pub hash: Hash32,
}

#[derive(Debug, Clone, PartialEq)]
pub enum PushError {
    AdmitError(ServedChainAdmitError),
    SenderUnavailable,
}

impl ServedChainHandle {
    pub fn new() -> (Self, ServedChainView);
    pub fn push_atomic(&self, accepted: AcceptedBlock)
        -> Result<ServedTip, PushError>;
    pub fn receiver_count(&self) -> usize;
}

impl ServedChainView {
    pub fn borrow(&self) -> watch::Ref<'_, ServedChainSnapshot>;
    pub fn subscribe(&self) -> Self;
}
```

`push_atomic` covers the full `served_chain_admit` call
inside a `watch::Sender::send_modify` closure — no observer
can read a torn snapshot mid-insertion (N11).

## §3 Doctrine recorded (DQ-B1 verbatim)

> Watch channel **guarantees**:
> - readers see a whole `ServedChainSnapshot` value
> - no torn push
> - delayed readers may skip intermediate snapshots
>
> Watch channel **does NOT guarantee**:
> - every peer observes every intermediate producer update
> - notification history

Block-fetch semantics depend on whether the requested block
is present in the snapshot read at dispatch time. Peers
requesting absent blocks receive `NoBlocks` per OQ8 (N4
fail-closed contract). If future multi-block serving needs
historical retention beyond the snapshot's own contents,
that history belongs inside `ServedChainSnapshot`, NOT inside
the transport handle.

## §4 Candidate registry entries (declared)

Three new entries proposed as `status = "declared"`. They
flip to `enforced` at B4 sub-cluster close.

### CN-SNAPSHOT-01 — push_atomic ordering + no-torn-snapshot

```toml
[[rules]]
id = "CN-SNAPSHOT-01"
tier = "derived"
statement = "A forged block becomes visible to peers only AFTER ServedChainHandle::push_atomic succeeds. The push_atomic call covers the full served_chain_admit call inside a watch::Sender::send_modify closure — no observer can read a torn snapshot mid-insertion. Coordinator emits BroadcastBlock; RED effect handler calls push_atomic; only on Ok(ServedTip) may per-peer reducers serve the block. Fail-closed shutdown on PushError."
```

### CN-SNAPSHOT-02 — partial-overlap RequestRange failure semantics

```toml
[[rules]]
id = "CN-SNAPSHOT-02"
tier = "derived"
statement = "A RequestRange covering a slot range that is not entirely present in ServedChainSnapshot MUST return NoBlocks per the Cardano block-fetch protocol's failure semantics — no partial ad-hoc response, no silent truncation, no serving of a strict prefix. Both endpoints + every block between MUST be present in the snapshot for the server to issue StartBatch + Block* + BatchDone."
```

### DC-SNAPSHOT-01 — push_atomic determinism

```toml
[[rules]]
id = "DC-SNAPSHOT-01"
tier = "derived"
statement = "ServedChainHandle::push_atomic is deterministic in its argument order: the same sequence of push_atomic(a₀), push_atomic(a₁), ..., push_atomic(aₙ) produces a byte-identical ServedChainView (fingerprint equals over BTreeMap insertion order). Replay-equivalence anchor for the broadcast → serve path."
```
