# Slice S3 — RED `producer_shell` (key custody + signing wrappers)

**Cluster:** PHASE4-N-Q
**Strengthens:** `CN-PROD-02` (key-custody half — RED owns secret
material; coordinator never sees keys), I7 (key custody
RED-confined).
**Introduces:** `crates/ade_runtime/src/producer/producer_shell.rs`
(RED).
**Dependencies:** PHASE4-N-Q S2 (HEAD `eb58794`).

## §1 Intent

Ship the RED-confined key-custody surface for live producer-mode.
`ProducerShell` holds the three signing keys (`KesSecret`,
`VrfSigningKey`, `ColdSigningKey`) and the operational
certificate, and exposes thin wrappers (`kes_sign_at`,
`vrf_prove`, `kes_advance_to`, `cold_vk`, `opcert`,
`public_metadata`). The GREEN coordinator gets the
`public_metadata` projection only; the keys never leave the
shell.

S3 explicitly does NOT compose the shell with `scheduler_step` /
`self_accept` / served-snapshot wiring — that's S5's main-loop
job. S3 is purely the type-confined key wrapper.

## §2 Scope

### Validation at init

- `opcert.hot_vkey.len() == 32` and `opcert.sigma.len() == 64`
  (wire-shape sanity).
- `kes_current_period ∈ [opcert.kes_period, opcert.kes_period + 63]`
  (KES key still covered by this opcert).

The full opcert-signature verification (cold-key signed) lives
in BLUE at `ade_core::consensus::opcert_validate` and is
invoked by the forge pipeline. S3's shell-init does not
re-validate that signature.

### Doctrinal carry-forward

- `KesSecret` / `VrfSigningKey` / `ColdSigningKey` keep their
  redacted `Debug`, no public byte accessors, drop-zeroize
  semantics (carried from N-O / N-P / N-C).
- `ProducerShell` itself has redacted `Debug`: secret fields
  print as `<redacted>`.
- No public byte accessor on the shell for any key.

### Out of scope (deferred)

- **S5:** `handle_request_forge(shell, request, ledger_inputs)
  -> CoordinatorEvent` orchestration that composes the shell
  with `scheduler_step` + `self_accept`. This requires
  resolving `LedgerSnapshotRef → LedgerState/ChainDep/Mempool`
  + assembling `TickInputs`; pure RED main-loop work.

## §4 Hard prohibitions

- **N9.** ProducerShell is the ONLY new RED surface that holds
  signing-key material. The coordinator (S2) does not import
  from this module (and the CI guard from S2 prevents it).
- No public byte accessor on any key wrapped by the shell.
- Custom `Debug` redaction on `ProducerShell` (not derived).
- Failure modes (`ShellInitError`, `ShellSignError`) carry
  only non-secret metadata.

## §5 Replay obligations preserved

S3 introduces no new replay obligations. The signing primitives
it wraps (`kes_sign`, `kes_update`, `vrf_prove`) are already
proven deterministic by PHASE4-N-C reference vectors +
PHASE4-N-P cross-impl + cardano-cli ground-truth corpus. The
shell wrappers add only the `kes_current_period` guard +
opcert-shape validation; no new bytes-shape commitments.
