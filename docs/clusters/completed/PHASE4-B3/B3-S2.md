### 3.0 Entry-proof obligation (HARD — no code before this lands)

Before any classification logic is implemented, enumerate Conway certificate tags `0..18` from
Conway CDDL / reference fixtures and produce a recorded table — committed as a fixture under
`corpus/conway_certs/classification_table.md` (+ machine-checkable per-tag fixtures) —
classifying **each** tag as exactly one of:

- `NewDeposit(coin)`
- `Refund(coin)`
- `Neutral`
- `NotValidInConway`

For every `NewDeposit`/`Refund` case, identify the **coin source**, exactly one of:
- **explicit in the certificate** (Conway explicit-deposit variants), or
- **sourced from canonical `ConwayDepositParams`** (legacy-implicit), or
- **sourced from ledger registration state** (e.g. pool reg new-vs-update, legacy dereg refund
  = deposit recorded at registration), or
- **unaccountable without state** → classified `UnsupportedStateDependentDepositAccounting`
  (a deterministic reject, never a guess).

Each row carries: tag, constructor name, deposit/refund-relevant encoded fields, effect, coin
source, state-independent vs state-dependent, and a reference citation / fixture proving the
mapping.

**Tags 5/6 are candidate `NotValidInConway`** (Shelley genesis-key-delegation / MIR, removed in
Conway) — **candidate, not proven** until the CDDL/reference check confirms they are absent
from the Conway cert grammar.

**No implementation branch may be accepted from a guessed table.** The candidate table in
`docs/clusters/PHASE4-B3/cluster.md` is planning input only; this fixture is the proven source,
and the totality test reads from it (never a hand-coded copy in test code).

## 4. Intent
Make it impossible for a Conway certificate to enter value-conservation accounting without an
explicit, proven disposition — every tag `0..18` resolves to a closed accounting effect, a
`Neutral` disposition, or a deterministic structured reject; an unknown/removed/malformed cert
can never be silently treated as `Neutral`. Introduces `DC-TXV-06`.

## 9. Design Summary

The accounting effect taxonomy is kept clean — an era-grammar reject is **not** a deposit
effect:

```rust
enum CertDisposition {
    Accountable(DepositEffect),
    Neutral,
    NotValidInConway,   // grammar / era-validity reject, NOT an accounting effect
}

enum DepositEffect {
    NewDeposit(CoinSource),
    Refund(CoinSource),
}

enum CoinSource {
    ExplicitInCert(Coin),     // Conway explicit-deposit variants
    DepositParam(Coin),       // legacy-implicit, from canonical ConwayDepositParams
    RegistrationState(Coin),  // resolved from ledger registration state
}
```

- Closed `ConwayCert` enum: every tag `0..18` is an explicit variant; the decoder has **no
  catch-all accept arm** — unknown tags → `CodecError::UnknownCertTag`, malformed CBOR →
  existing structured `CodecError`.
- `classify` is total over `ConwayCert` (compiler-enforced exhaustiveness). Tags 5/6 →
  `CertDisposition::NotValidInConway` (final disposition pinned by §3.0). State-dependent
  variants that cannot be accounted from available state return
  `UnsupportedStateDependentDepositAccounting` rather than guessing.
- Three clean failure classes, never conflated:
  1. **decode failure** — `CodecError` (unknown tag / malformed CBOR),
  2. **era-validity failure** — `CertDisposition::NotValidInConway` (known tag, removed in Conway),
  3. **accounting-environment insufficiency** — `UnsupportedStateDependentDepositAccounting`.

## 13. Failure Modes
Three distinct classes, never conflated:
- **Decode failure** — `CodecError::UnknownCertTag { tag, offset }` (unknown/undefined tag) or
  existing `CodecError::{InvalidCborStructure, UnexpectedEof, ...}` (malformed CBOR). Fail-fast,
  deterministic; never coerced to a known variant.
- **Era-validity failure** — `CertDisposition::NotValidInConway` for a known-but-removed tag
  (5/6, per §3.0). Consumed by B3-S4 as a structured Invalid verdict; **not** an accounting effect.
- **Accounting-environment insufficiency** — `UnsupportedStateDependentDepositAccounting` when a
  state-dependent effect cannot be resolved from available state. Deterministic reject, distinct
  from both a malformed-cert error and an era-validity reject.

All three are fail-fast and replay-deterministic: same input → same class.

All "Forbidden During This Cluster" items from `docs/clusters/PHASE4-B3/cluster.md` — notably:
classifying an unknown/malformed cert as `Neutral`; decoding Conway certs with the 6-variant

### Slice-Specific Prohibitions
- Any catch-all `_ =>` arm in the Conway decoder that *accepts* (the Shelley decoder's
  `StakeRegistration(zero-hash)` fallback must not be reproduced).
- Modeling `NotValidInConway` as a `DepositEffect` variant (era-validity is not an accounting effect).
- Conflating the three failure classes (decode vs era-validity vs accounting-environment).
- Classifying a state-dependent case by guessing instead of `UnsupportedStateDependentDepositAccounting`.
- Hand-coding the tag→effect table in test code instead of reading the committed §3.0 fixture.
- Modifying the Shelley decoder or `rules.rs:1419-1440` cert path (PHASE4-B4 scope).
- `String`/`anyhow`/formatted errors in the BLUE decode/classify path.

**If any of these appear, the slice is incorrect.**
