## 3a. Slice-Entry Proof Obligations (gating)

These compatibility facts gate implementation; they are not footnotes. If any fails, the
design changes.

- **PO-1 (era equivalence).** Conway CDDL `pool_registration_cert` field order and shape
  for `{ operator/pool_id, vrf_keyhash, pledge, cost, margin, reward_account,
  pool_owners }` must be byte/semantics-identical to the Shelley `pool_registration_cert`.
  Expectation: identical (cardano-ledger keeps `pool_params` era-stable). **Must be
  confirmed against the Conway CDDL, not assumed.** If Conway diverges, shared-type reuse
  is **forbidden** → introduce a Conway-specific pool-registration type.
- **PO-2 (owner-completeness).** The retained pool-param field set must equal exactly what
  the authoritative owner `PoolParams` stores. Relays and `pool_metadata` are excluded
  *because* `PoolParams` has no such field. Re-confirm `PoolParams`' shape
  (`crates/ade_ledger/src/…`) at implementation so the exclusion stays grounded; if
  `PoolParams` gains relays/metadata, they must be retained too.
- **PO-3 (fingerprint neutrality).** B4-S1 leaves `apply_pool_registration` (and all ledger
  apply) untouched, so `PoolState` and the replay fingerprint are unchanged by this slice.
  The newly-retained `owners` are decoded-and-tested but not yet consumed (B4-S2 consumes
  them and removes the `owners: Vec::new() // TODO` stub).

## 4. Intent
Make it impossible for the single authoritative Conway certificate decoder to be a *lossy
projection*: every certificate (tags 0..18) must decode into a `ConwayCert` carrying **all
fields any authoritative owner needs** (delegation, pool incl. owners, DRep, committee,
vote-delegation), not merely B3's deposit/refund fields. This closes the finer-grained
instance of the Shelley-decoder bug class — payload-dropping inside `ConwayCert` (and
inside the shared `PoolRegistrationCert`) — that would otherwise force B4-S2's
`apply_conway_cert` to mutate state from data the decoder threw away.

## 5. Scope
- **Modules / crates:**
  - **BLUE** `ade_types::conway::cert` — enrich `ConwayCert` variants with owner-payload fields (§9 table).
  - **BLUE** `ade_types::shelley::cert::PoolRegistrationCert` — add `owners: Vec<Hash28>`
    (single authoritative pool-reg type; reused and enriched, never duplicated).
  - **BLUE** `ade_codec::conway::cert` — decode the currently-skipped/discarded payloads;
    add `decode_drep`; use the shared pool-param helper for tag 3.
  - **BLUE** `ade_codec::shelley::cert` — decode `pool_owners` into the shared
    `PoolRegistrationCert` (currently skipped); since the type is shared, the field must be
    populated wherever the type is produced. Factor a shared `decode_pool_registration_params`.
  - **BLUE** `ade_ledger::cert_classify` — update `classify` match arms (and its unit-test
    cert constructors) to `{ field, .. }` so the B3 deposit projection still compiles and
    yields **byte-identical** `CertDisposition`.
- **State machines affected:** none (decoder is pure; unwired to ledger apply — that is B4-S3).
- **Persistence impact:** none. `ConwayCert`/`PoolRegistrationCert` are in-memory decode
  products, not persisted/fingerprinted. No WAL/checkpoint change. (See PO-3.)
- **Network-visible impact:** none.
- **Out of scope (explicit):**
  - `apply_conway_cert` / `ConwayCertAction` / cert-state mutation (B4-S2).
  - Wiring the decoder into `process_block_certificates` / era dispatch (B4-S3).
  - The apply-side `owners` stub (`apply_pool_registration` TODO) — B4-S1 *retains* owners
    in the decoder; B4-S2 *consumes* them and removes the TODO.
  - Pool **relays** and **pool_metadata** — no authoritative owner stores them; structurally consumed.
  - Certificate **anchors** (metadata URL+hash) — no owner stores them; structurally consumed.
  - Enriching `StakeCredential` with its key-vs-script discriminator — the 28-byte hash is
    collision-free across key/script preimages, so hash-keyed owner state is observationally
    equivalent; not a B4 obligation.

## 9. Design Summary
Additive enrichment of a closed sum type plus the decode logic to populate it.

**Enriched `ConwayCert` fields (by tag):**

| tag | variant | fields after this slice |
|---|---|---|
| 0 | AccountRegistration | `credential` *(unchanged)* |
| 1 | AccountUnregistration | `credential` *(unchanged)* |
| 2 | StakeDelegation | **+`credential`, +`pool_id`** |
| 3 | PoolRegistration | **`PoolRegistrationCert`** — full retained params `{ pool_id, vrf_hash, pledge, cost, margin, reward_account, owners }`; `pool_id` derived from the cert |
| 4 | PoolRetirement | **+`pool_id`, +`epoch`** |
| 5/6 | RemovedInConway | `tag` *(unchanged)* |
| 7 | AccountRegistrationDeposit | **+`credential`**, `deposit` |
| 8 | AccountUnregistrationDeposit | **+`credential`**, `refund` |
| 9 | VoteDelegation | **+`credential`, +`drep`** |
| 10 | StakeVoteDelegation | **+`credential`, +`pool_id`, +`drep`** |
| 11 | StakeRegistrationDelegation | **+`credential`, +`pool_id`**, `deposit` |
| 12 | VoteRegistrationDelegation | **+`credential`, +`drep`**, `deposit` |
| 13 | StakeVoteRegistrationDelegation | **+`credential`, +`pool_id`, +`drep`**, `deposit` |
| 14 | AuthCommitteeHot | **+`cold_credential`, +`hot_credential`** |
| 15 | ResignCommitteeCold | **+`cold_credential`** (anchor consumed, not retained) |
| 16 | DRepRegistration | **+`drep_credential`**, `deposit` (anchor consumed) |
| 17 | DRepUnregistration | **+`drep_credential`**, `refund` |
| 18 | DRepUpdate | **+`drep_credential`** (anchor consumed) |

**Decode mechanics:** replace each `_credential`/`_pool`/`skip_drep` discard with a retained
decode; decode the tags that currently read nothing (2/9/10/14/15/18) instead of relying on
the trailing `skip_item`. Add `decode_drep` (CDDL `drep = [0 keyhash] / [1 scripthash] /
[2] / [3]` → existing `DRep` enum). Tag 3 uses the shared `decode_pool_registration_params`
(extracted from the Shelley arm) producing the enriched `PoolRegistrationCert`. The trailing
`*offset = cert_start; skip_item` normalisation stays — it still consumes any
anchor/relays/metadata tail, preserving trailing-bytes strictness.

**Projection preservation:** `cert_classify::classify` arms that bind fields gain `, ..`;
the returned `CertDisposition` is byte-identical. Its in-module test constructors are updated
to the enriched variant shapes.

## 13. Failure Modes
- **Malformed cert CBOR** (wrong array shape, non-28-byte hash, missing field for a
  newly-decoded payload, bad pool-owner list) → `CodecError::{InvalidCborStructure,
  InvalidLength, UnexpectedEof}`; fail-fast; rejects the whole cert array (no partial
  accept). Affects replay only by rejecting input previously skipped-over — surfaced, never
  silently consumed.
- **Unknown tag ≥19** → `CodecError::UnknownCertTag`; fail-fast (unchanged).
- **Removed tag 5/6** → `ConwayCert::RemovedInConway` (unchanged; era-validity reject is downstream).
- **Trailing bytes after cert array** → `CodecError::TrailingBytes` (unchanged).
- **Malformed DRep** (unknown DRep variant tag) → structured `CodecError`; fail-fast.

All failure shapes are deterministic `CodecError` variants → replay-safe.

## 14. Hard Prohibitions
### Inherited Cluster-Level Prohibitions
All of PHASE4-B4 "Forbidden during this cluster" — in particular: no second Conway cert
decoder; no payload-dropping inside `ConwayCert` (or inside the shared `PoolRegistrationCert`
for fields an owner needs); no reduction into the Shelley `Certificate`; no
`HashMap`/float/clock in BLUE; no re-encode for any hash.

### Slice-Specific Prohibitions
- No `apply_conway_cert`, cert-state mutation, or `process_block_certificates` change (later slices).
- No removal of the `apply_pool_registration` `owners` TODO (that is B4-S2; B4-S1 only retains the data).
- No new owner-payload type when an authoritative one exists; introduce a Conway-specific
  pool-reg type **only** if PO-1 fails.
- No catch-all `_ =>` accept arm in the decoder; no `_ =>` wildcard added to `cert_classify::classify`.
- No relaxing the B3F trailing-bytes / preallocation-bound checks.
- No anchor / relays / metadata retention; no `StakeCredential` shape change.
- No behavior change to any `CertDisposition` returned by `classify`, and no change to
  `PoolState`/fingerprint (PO-3).
