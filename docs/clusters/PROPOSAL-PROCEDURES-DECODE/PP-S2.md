## Intent

Provide a GREEN replay harness over a canonical corpus of Conway
`proposal_procedures` byte sequences. Asserts that decoding produces
the expected typed shape and re-encoding produces byte-identical
output. Per OQ-5, the existing in-tree Conway corpus contains no txs
carrying `proposal_procedures` (verified at HEAD), so the corpus is
**synthetic canonical fixtures** generated by the harness itself —
a legitimate close, not a deferral. When real-chain bytes become
available later, the harness can be extended in a separate
strengthening slice.

```rust
// crates/ade_testkit/src/governance/proposal_procedures_replay.rs

/// One entry in the canonical proposal_procedures corpus: a labeled
/// pre-encoded byte sequence and the typed shape it MUST decode to.
pub struct CorpusEntry {
    pub label: &'static str,
    pub bytes: Vec<u8>,
    pub expected: Vec<ProposalProcedure>,
}

/// Outcome of replaying one corpus entry through the closed decoder.
#[derive(Debug, PartialEq, Eq)]
pub enum ReplayOutcome {
    Ok,
    DecodeMismatch { label: &'static str },
    EncodeMismatch { label: &'static str },
    DecodeError { label: &'static str, error: String },
}

/// The canonical corpus: one entry per GovAction variant (covering
/// the structural shapes that round-trip is load-bearing for), plus a
/// multi-procedure mixed-variant entry, plus a corner case
/// (single-procedure InfoAction — the smallest valid set).
pub fn canonical_corpus() -> Vec<CorpusEntry>;

/// Replay every corpus entry through `decode_proposal_procedures` and
/// `encode_proposal_procedures`. Asserts:
///   - decode succeeds and produces `expected`,
///   - re-encode is byte-identical to the original `bytes`.
/// Returns `ReplayOutcome::Ok` on success or the diagnostic variant.
pub fn replay_canonical_corpus() -> Vec<(String, ReplayOutcome)>;
```

The corpus generator produces canonical CBOR via the BLUE
`encode_proposal_procedures` (so the bytes are themselves canonical
by construction). The replay then decodes them, asserts the typed
shape, and re-encodes to confirm byte-identicality. This is a
slightly tautological loop for the encode-decode-encode case, but
it's the load-bearing property: the harness IS the corpus, and
adding real-chain corpus later is purely additive.

### 2. New integration test
   `crates/ade_testkit/tests/proposal_procedures_corpus_replay.rs`

  - `canonical_corpus_decodes_to_expected_shape` — for every entry,
    decoding produces the typed value the entry declares as
    `expected`.
  - `canonical_corpus_round_trips_byte_identical` — for every entry,
    `encode(decode(bytes))` equals the original `bytes`.
  - `canonical_corpus_covers_all_gov_action_variants` — the corpus

  - `canonical_corpus_includes_update_committee_discriminant_case` —
    at least one entry has `UpdateCommittee` with KeyHash + ScriptHash
    of equal 28 bytes (DC-LEDGER-10 cross-check in the corpus).

## Hard Prohibitions

- No new registry rule.
- No new CI script (the harness is evidence, not enforcement).
- No production code changes (harness is GREEN, testkit-only).
- No reuse of real-chain corpus that does not exist (the synthetic
  fallback per OQ-5 is the agreed close).
- No `voting_procedures` / `ParameterChange.update` /
  `NewConstitution.raw` / typed `RewardAccount` work (cluster scope
  locks).
