# S4.2 — consume the valid V2 store: thread the DRep/committee authority into the live gate

## The narrow question

S4.1b produced a valid, source-bound V2 governance seed. S4.2 asks ONE thing: does the live path correctly
consume it — i.e., when the imported authority is threaded into the live gate, does the gate reproduce the
oracle? Not "does the whole cluster work" (that's S4.3/S5/S6); just: valid V2 state in → correct gate out.

## Gate analysis (what is safe to thread now)

`mithril_native_assembly.rs` seeds the live `ConwayGovState` with the gate fields EMPTY (import-not-activate).
The three gates behave differently under threading:

- **Committee** (`check_ratification` ~234): already active (committee + quorum are seeded); `committee_hot_keys`
  is empty so hot-votes fall back to counted. Threading `committee_hot_keys` makes hot→cold resolution exact.
- **DRep** (~276): INERT (empty `drep_voting_thresholds` + empty `vote_delegations` ⇒ `total_drep_active_stake
  = 0` ⇒ skipped). The gate is `yes_stake / total_active` — **MONOTONE**: a No/absent DRep merely stays in the
  denominator (never inflates it beyond the fixed active total), so a proposal it rejects is genuinely
  unratifiable-via-DRep. Threading `vote_delegations` + `drep_voting_thresholds` + `drep_expiry` (+ the already
  Bound `num_dormant`) activates it SAFELY.
- **SPO** (~325): `voted_stake = yes + no` — **NON-MONOTONE**. A captured No inflates the denominator and can
  flip a borderline proposal to "unratifiable" → with the current CPDE terminal that expired proposal would be
  REFUNDED — a wrongful refund. So `pool_voting_thresholds` MUST NOT be threaded until S4.3 replaces the
  terminal with oracle-verified ratify-then-enact.

## S4.2 scope

Thread from `imported_gov` into the live `mithril_native_assembly` seeding: `vote_delegations`, `drep_expiry`,
`drep_voting_thresholds`, `committee_hot_keys`. LEAVE `pool_voting_thresholds` EMPTY (SPO stays inert → S4.3).
`num_dormant` is already `Bound` (S4.1b). The accumulator already derives `drep_stake` via
`derive_drep_voting_stake` (S3) and passes `num_dormant` (S4.1) — so once the fields are non-empty the DRep
gate fires with real inputs.

## Invariants / gates (CE)

1. **No CPDE regression.** Threading the DRep gate MUST NOT turn a CE-3d expiry-refund into a
   potentially-ratifiable HALT or a wrongful refund. The CE-3d window (1340–1342) is expiry-only; verify the
   cpde_s4_0 / cpde_s5 refund set is unchanged (the CE-3d proposals must stay provably-unratifiable, i.e. they
   lack the DRep-Yes stake to become "potentially ratifiable"). If any flips, S4.2 has changed authoritative
   behavior and must be reconciled against the oracle before landing.
2. **Oracle-verified DRep consumption.** The live gate, fed the threaded authority, reproduces the census DRep
   decision — the c2f5960e anchor already proves `evaluate_ratification` ratifies 69c948cd..#0 @1095 with the
   real DRep stake; S4.2 proves the LIVE seeding delivers that same authority to the gate.
3. **SPO stays inert.** `pool_voting_thresholds` empty ⇒ the non-monotone SPO gate cannot drive a refund. The
   no-wrongful-refund invariant holds by construction until S4.3.
4. **Determinism + import-not-activate lineage.** The threaded fields come from the commitment-bound
   `ImportedGovState` (v10) — the same named source; no fabrication.

## What S4.2 does NOT do (S4.3)

The `PotentiallyRatifiable` terminal is UNCHANGED in S4.2 — so the node still HALTS on a DRep/committee-
ratifiable proposal (the CPDE continuity blocker persists). S4.2 makes the gate DETERMINATION accurate + the
DRep authority live; S4.3 threads the SPO threshold AND replaces the terminal with atomic ratify-then-enact
(a ratified proposal takes the ENACT path, never the refund path), resolving the continuity blocker.
