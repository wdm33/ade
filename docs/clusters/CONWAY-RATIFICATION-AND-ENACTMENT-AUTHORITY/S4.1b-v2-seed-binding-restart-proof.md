# S4.1b — V2 governance seed binding and restart proof

## Why this is its own slice (not folded into S4.2)

Producing and persisting a V2-bound governance seed is an **authoritative state-lineage change** (the
download/import action is RED, but the decoded→bound→persisted→recovered `num_dormant` is BLUE authority). It
deserves its own proof boundary, sequenced BEFORE the S4.2 live flip. If re-bootstrap and activation land
together, a failed anchor rerun has TWO plausible causes — a bad V2 seed (decode / bind / persist / recover)
or a live gate that consumes correct V2 state incorrectly. Those are different authority surfaces; keep them
independently diagnosable. S4.1b proves the seed is correct; S4.2 then narrows to "given valid V2 state, does
the live path consume it correctly?"

## What the machinery already gives us (from S4.1)

The durable gov-state persistence already routes through the VERSIONED codec: `snapshot/ledger.rs:68/104` and
`epoch_accumulator.rs:1516/1566` call `encode_gov_state`/`decode_gov_state`, which emit 9 fields for
`Unversioned` (V1, byte-identical) and 10 for `Bound(n)` (V2). The bootstrap already binds
`Bound(s1a.imported_gov.num_dormant_epochs)` (`mithril_native_assembly.rs:389`), decoded fail-closed from the
VState (`read_vstate` → `read_u64(vsNumDormantEpochs)`). So S4.1b is largely PROOF that the round-trip binds,
persists, and recovers the real value — plus the legacy negative fixture.

## Hard rules

- Create a NEW, SEPARATE V2 store from verified source material — do NOT mutate the old CE-3d `Unversioned`
  store in place.
- Bind `num_dormant_epochs` DIRECTLY to the decoded `vsNumDormantEpochs` — never a fabricated 0 / `Default` /
  inferred value.
- Persist the V2 state, warm-restart it, and prove the SAME V2 value AND fingerprint reappear.
- Keep the legacy `Unversioned` store as a NEGATIVE fixture: a DRep-expiry-dependent path over it must still
  produce `DormantStateRequired` (the fail-closed refuses to invent the offset).
- Use the same named bootstrap anchor / source point as the CE-3d corpus where possible; if a different
  verified point is unavoidable, record it as a SEPARATE evidence lineage, not silently the same seed.

## The proof is NOT "old and new fingerprints match"

V1 and V2 encode DIFFERENT authoritative state — their fingerprints SHOULD differ. The required proof is:
the V2 seed is **source-bound**, **replay-stable across restart**, carries the **real dormant offset**, and
yields the **same oracle ratification outcomes** for the anchor corpus.

## Acceptance gate (all required before S4.2 begins)

1. **Verified bootstrap binding** — manifest / network / era / anchor coherence holds for the new store
   (the existing native-assembly coherence gate), and the assembled gov state's `num_dormant` is
   `Bound(imported.num_dormant_epochs)`.
2. **Decoded `vsNumDormantEpochs` present and non-fabricated** — sourced from the real VState decode, exercised
   with a NON-ZERO value so a laundered 0 cannot pass.
3. **V2 encode → decode → fingerprint stability** — a `Bound(n)` gov state round-trips through the DURABLE
   path (accumulator-store / ledger snapshot), preserving `n` and the fingerprint.
4. **Durable restart equality** — persist a V2 store, recover it, and assert the recovered gov state is equal
   (value + governance fingerprint) to the persisted one.
5. **Legacy-`Unversioned` fail-closed regression** — an `Unversioned` store on a DRep-expiry-dependent path
   still yields `DormantStateRequired` (kept as a negative fixture, not migrated).
6. **`c2f5960e` anchor rerun against the V2 store** — the recovered V2 governance state, fed to the ratify
   gate, reproduces the oracle: 69c948cd..#0 ratifies at 1095, not at 1094, and is the only proposal
   ratified (no false positives), with the expected V2 fingerprints where V2 is used.

## Then

- **S4.2** — the narrow live flip: thread the imported authority (incl. the V2 `num_dormant`) into the live
  `mithril_native_assembly` gov_state seeding and the active gate; prove the LIVE path consumes the valid V2
  store correctly (the anchor holds on the live path).
- **S4.3** — replace the `PotentiallyRatifiable` terminal with atomic ratify-then-enact; a ratified proposal
  takes the ENACT path, never the refund path (SPO-non-monotonicity: a captured SPO No cannot wrongful-refund).
