## §3 Branch coverage analysis

The CN-FORGE-01 contract names 4 variants:

| Variant | A4 coverage | Mechanism |
|---|---|---|
| `ForgeNotLeader` | ✅ Direct integration test | Zero-stake answer at step 2 |
| `ForgeFailed { KesPeriodMismatch }` | ✅ Direct integration test | `u32::MAX` period at step 3 |
| `ForgeFailed { SelfAcceptRejected }` | ✅ Direct integration test (or earlier `Other`/`EmptyMempool` from `forge_block`'s opcert / mempool checks) | Full-stake answer reaches step 5/6, synthetic block rejected |
| `ForgeSucceeded` | ⚠ Structural guarantee only | The closed pipeline has exactly one path to emit `Succeeded` (past `self_accept = Accepted` at step 6). N-R-A's A4 test 3 explicitly panics if `ForgeSucceeded` is reached, asserting the gate holds under honest scope. End-to-end ForgeSucceeded testing requires the KES-signs-real-unsigned-header bridge — a future post-N-R cluster. |

This split is documented in the `CN-FORGE-01.evidence_notes`
and `CN-FORGE-01.open_obligation` fields. The structural
property "no `ForgeSucceeded` without `self_accept = Accepted`"
is enforced today by the closed pipeline shape.
