## §2 Pre-flight proof obligations — resolutions

### OQ4 — opcert envelope shape

**Captured** against docker `cardano-node-preprod` running
cardano-node 11.0.1 with cardano-cli 11.0.0.0 git rev
`97036a66bcf8c89f687ae57a048eecc0389977ef`.

**Locked envelope shape:**

```
{
  "type": "NodeOperationalCertificate",
  "description": "",
  "cborHex": <hex of CBOR>
}
```

The `cborHex` decodes to CBOR `array(2)`:

- **Element 0:** `array(4)` of `[hot_vkey(32-byte bytes),
  sequence_number(uint), kes_period(uint),
  sigma(64-byte bytes)]`.
- **Element 1:** `bytes(32)` — `cold_vk` (Ed25519 issuer
  verification key for sigma verification).

The existing `ade_types::shelley::block::OperationalCert`
struct matches element 0 directly. N-R-C C1's parser
extracts element 0 + optionally element 1 (for sigma
self-verification).

**Fixtures at `crates/ade_runtime/tests/fixtures/opcert/`:**
- `accepted-cardano-cli-11.0.0.opcert.json` — real
  well-formed envelope (parser MUST accept).
- `kes.vkey.json` + `cold.vkey.json` + `cold.counter.json`
  — companion envelopes (the KES VK matches opcert
  element 0[0] byte-for-byte; the cold VK matches opcert
  element 1 byte-for-byte).
- `malformed-type.opcert.json` — wrong envelope type
  (parser MUST reject).
- `malformed-cborhex.opcert.json` — non-hex in cborHex
  (parser MUST reject).
- `wrong-arity.opcert.json` — inner array has 5 elements
  (parser MUST reject).

See `crates/ade_runtime/tests/fixtures/opcert/FIXTURE_METADATA.md`
for the full capture procedure, decoded field values, and
use sites.

### OQ7 — Conway genesis closed parser contract

**Captured** from the docker container's bind-mounted
`.cardano-node-preprod/config/` directory.

**Important finding:** the load-bearing slot/KES geometry
fields live in **Shelley** genesis, not Conway. Conway
genesis carries governance parameters only. The C2 parser
is therefore a **Shelley-genesis-+-Conway-addendum parser**:

| `GenesisAnchor` field | Source key in `shelley-genesis.json` | Type |
|---|---|---|
| `network_magic` | `networkMagic` | u32 |
| `slot_zero_time_unix_ms` | `systemStart` (ISO 8601, parsed deterministically) | u64 |
| `slot_length_ms` | `slotLength` (seconds, × 1000) | u64 |
| `slots_per_kes_period` | `slotsPerKESPeriod` | u64 |
| `kes_max_period` | `maxKESEvolutions` | u32 |
| `kes_anchor_slot` | operator-supplied (NOT in genesis) | u64 |

**Hard rules (carry-forward from DQ-C2):**

- Required fields fail-closed on missing / malformed /
  wrong-type input.
- No implicit defaults ("if missing, assume preprod"
  rejected).
- No stringly fallback (`"1"` rejected for u32 fields).
- Extra keys accepted-and-ignored for forward compatibility,
  iff inert. The `GenesisAnchor` produced by an
  extra-key fixture MUST byte-equal the `GenesisAnchor`
  produced by the canonical fixture.

**Fixtures at `crates/ade_runtime/tests/fixtures/conway_genesis/`:**
- `accepted-shelley-genesis.json` — real preprod
  Shelley genesis (parser MUST accept).
- `accepted-conway-genesis.json` — real preprod Conway
  genesis governance addendum (parser MUST accept;
  fields not consumed by `GenesisAnchor`).
- `missing-required.shelley-genesis.json` — `networkMagic`
  missing (parser MUST reject).
- `stringly-int.shelley-genesis.json` — `"1"` instead of
  `1` (parser MUST reject; no stringly fallback per DQ-C2).
- `extra-inert-key.shelley-genesis.json` — extra unknown
  keys (parser MUST accept and produce byte-identical
  `GenesisAnchor`).
- `malformed-numeric.shelley-genesis.json` — `slotLength:
  -1` (parser MUST reject).

See `crates/ade_runtime/tests/fixtures/conway_genesis/FIXTURE_METADATA.md`
for the full mapping table, real preprod values, and use
sites.

### OQ8 — block-fetch protocol failure semantics

**Resolution from existing code audit.** The Ade
block-fetch server (`crates/ade_network/src/block_fetch/server.rs`)
currently handles `RequestRange` as follows (lines 149-184):

| Input | Reply |
|---|---|
| `RequestRange { from: Origin, .. }` | `NoBlocks` (single reply) |
| `RequestRange { .., to: Origin }` | `NoBlocks` |
| `RequestRange { from, to }` where `served.range_bytes(from, to)` is empty | `NoBlocks` |
| `RequestRange { from, to }` where the BTreeMap range is non-empty | `[StartBatch, Block(b)*, BatchDone]` — returns *whatever is in the range* |

**Finding:** the current Ade behavior is **non-compliant**
for the partial-overlap case. Specifically, if `from`
exists in the snapshot but `to` does not (or vice-versa),
the current implementation returns whatever blocks the
BTreeMap's `range(from..=to)` iterator yields, which is
a strict subset — not the protocol-defined `NoBlocks`
response for an unknown / partially-unavailable range.

Per the `ouroboros-network` typed-protocols reference
(Haskell), the canonical cardano-node behavior is:

- If the server has **all** requested blocks in the range
  (both endpoints present + every block between) → reply
  `StartBatch + Block* + BatchDone`.
- If the server is missing the range (endpoint not found
  OR any intermediate block absent) → reply `NoBlocks`.

**This is a real N-R-B bug fix, not just a test.** B4's
partial-overlap test corpus MUST include:

1. `RequestRange` with `from ∉ snapshot, to ∈ snapshot` →
   expect `NoBlocks` (currently fails — returns partial).
2. `RequestRange` with `from ∈ snapshot, to ∉ snapshot` →
   expect `NoBlocks` (currently fails).
3. `RequestRange { from, to }` both present but with a
   missing intermediate slot (sparse snapshot) → expect
   `NoBlocks` (currently fails — returns the present
   blocks only).

B2 or B4 modifies `producer_block_fetch_serve` to: require
both endpoints present in the snapshot AND assert the range
is contiguous before issuing replies. Otherwise → `NoBlocks`.

### OQ9 — `dispatch_chain_sync_frame` / `dispatch_block_fetch_frame` signatures

**Resolution from existing code audit.** The dispatch API
in `crates/ade_runtime/src/network/n2n_server.rs`:

```rust
pub fn dispatch_chain_sync_frame(
    state: PerPeerN2nServerState,
    frame: &[u8],
    snap: &ServedChainSnapshot,
) -> Result<(PerPeerN2nServerState, Option<Vec<u8>>, bool), DispatchError>;

pub fn dispatch_block_fetch_frame(
    state: PerPeerN2nServerState,
    frame: &[u8],
    snap: &ServedChainSnapshot,
) -> Result<(PerPeerN2nServerState, Vec<Vec<u8>>, bool), DispatchError>;

pub fn poll_chain_sync_advance(
    state: PerPeerN2nServerState,
    snap: &ServedChainSnapshot,
) -> Result<(PerPeerN2nServerState, Option<Vec<u8>>), DispatchError>;
```

**Finding:** the dispatch functions take `snap: &ServedChainSnapshot`
**by reference** (not owned, not `&dyn Trait`). This is the
ideal shape for N-R-B's
`tokio::sync::watch::channel<ServedChainSnapshot>` pattern
(DQ-B1):

- The per-peer task receives the latest `ServedChainSnapshot`
  via `watch::Receiver::borrow()`, which returns
  `Ref<'_, ServedChainSnapshot>`.
- The peer task then calls `dispatch_*(state, frame, &*snap_ref)`
  with the dereferenced borrow.
- The borrow holds the watch read lock for the duration of
  the dispatch call; no torn snapshot is possible (N11).

**No dispatch API refactor needed.** B2 + B3 wire the
watch-channel handle to the existing dispatch entry points
directly.

The `bool` element of the chain-sync return tuple is
"session terminated (client sent Done)"; B3 routes this to
a `PeerDisconnect { reason: ClientDone }` event.
