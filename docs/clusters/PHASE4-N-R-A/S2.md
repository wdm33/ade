## §1 Intent

Land the BLUE leader-check evaluator + relocate the
canonical `is_leader_for_vrf_output` helper so leader
eligibility is a callable, closed surface — not a buried
step inside `forge_block`.

The new module `ade_core::consensus::leader_check`:

- **`verify_and_evaluate_leader`** — narrow BLUE evaluator
  taking caller-provided `LeaderScheduleAnswer` + RED-issued
  VRF proof; returns closed two-variant `LeaderCheckVerdict`.
- **`LeaderCheckVerdict`** — closed enum:
  - `Eligible { slot, vrf_output, leader_proof }` carries
    forge-capable material.
  - `NotEligible { slot, vrf_output_fingerprint }` carries
    only non-forge-capable evidence.
- **`is_leader_for_vrf_output`** — relocated from
  `leader_schedule.rs`; remains `pub fn` for the existing
  forge.rs defense-in-depth pin (NC-VRF-3 single source of
  leader truth); new external callers blocked by CI gate.

## §3 BLUE-isolation properties

`verify_and_evaluate_leader` has **no** dependency on:

- `LedgerView` (caller derives `LeaderScheduleAnswer` via
  `query_leader_schedule`).
- `EraSchedule` (same).
- `PraosChainDepState` (caller passes `eta0: &Nonce`
  directly; `query_leader_schedule` is the conversion).
- Wall-clock time (no timestamps, no `Instant::now()`).
- Storage / file I/O.
- RED crates (`ade_runtime`, `tokio`, etc.).

This is the **CE-A-6b** exit gate from the cluster doc.
The grep gate enforces the constraint by allow-list (no
new callers of `is_leader_for_vrf_output` outside the
permitted set).

## §4 LeaderCheckVerdict shape (final)

```rust
pub enum LeaderCheckVerdict {
    Eligible {
        slot: SlotNo,
        vrf_output: VrfOutput,           // forge-capable
        leader_proof: LeaderProofFingerprint, // audit binding
    },
    NotEligible {
        slot: SlotNo,
        vrf_output_fingerprint: VrfOutputFingerprint, // bounded evidence
    },
}
```

**Structural property:** `NotEligible` does **not** carry
`vrf_output`. The closed enum shape makes it impossible for a
non-leader-handling caller to observe forge-capable material
under the type system — a `match` over the verdict cannot
extract `vrf_output` from `NotEligible`.

## §5 Error surface

```rust
pub enum LeaderCheckError {
    VrfInputMismatch { expected_first_8: [u8; 8], answer_first_8: [u8; 8] },
    AnswerSlotMismatch { function_slot: u64, answer_slot: u64 },
    VrfVerificationFailed,
    ZeroStakeDenominator,
}
```

All variants are structured (no `String` payloads). The
first-8-byte fingerprints in `VrfInputMismatch` are
diagnostic only — they may legitimately be equal if the
mismatch is in bytes 8..40 (the epoch_nonce portion), since
bytes 0..8 encode the slot. Callers MUST NOT branch on
fingerprint equality; the variant existing is the gate.
