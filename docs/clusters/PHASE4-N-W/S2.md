# PHASE4-N-W S2 — Praos producer leader-VRF migration

> Slice of cluster PHASE4-N-W. Addresses **CE-W-1, CE-W-2, CE-W-3, CE-W-4,
> CE-W-5, CE-W-7, CE-W-8**. The atomic semantic change — the producer
> leader-VRF alpha must agree end-to-end, so the type + the single authority +
> the three producer sites move together.

## Invariant scope

For a Praos era the producer's leader-eligibility VRF uses the same construction
the validator runs: alpha `praos_vrf_input(slot, eta0)`, threshold over
`praos_leader_value(output)`, single combined proof. One shared BLUE
era→construction authority feeds the leader-schedule (build), the leader-check
evaluator (verify), and the forge prove-step (no independent re-derivation). The
era is carried in the alpha *type* so a TPraos alpha in a Praos context is
unrepresentable (N4). (I1, I2, I3, N3, N4.)

Grounded facts (verified): the validator already era-discriminates via
`HeaderVrf::{Tpraos, Praos}`; the forge already emits `VrfData::Combined →
HeaderVrf::Praos`; `forge_succeeds.rs` fails *only* at the proof-verify step
(wrong alpha). So the semantic fix is the alpha + the threshold derivation, not
the header structure.

## Changes

### BLUE authority — `ade_core::consensus::vrf_cert`
- New `enum ExpectedVrfInput { Praos([u8;32]), Tpraos([u8;VRF_INPUT_LEN]) }` +
  `alpha_bytes()`. The variant IS the protocol-family tag (N4).
- `leader_vrf_input(era, slot, eta0) -> ExpectedVrfInput` — the single
  era→leader-VRF-input authority (N3). Praos → `praos_vrf_input`; else →
  role-tagged `vrf_input`.
- `leader_value_for(&ExpectedVrfInput, &VrfOutput) -> VrfOutput` — Praos →
  `praos_leader_value`; TPraos → raw output (clone). Era family read from the
  variant.
- Re-export the three from `consensus::mod`.

### BLUE — `leader_schedule`
- `LeaderScheduleAnswer.expected_vrf_input: [u8;VRF_INPUT_LEN]` →
  `ExpectedVrfInput`.
- `query_leader_schedule` builds via `leader_vrf_input(location.era, …)` (it
  already locates the era).

### BLUE — `leader_check`
- `verify_and_evaluate_leader` gains `era: CardanoEra` (first param). Rebuilds
  `leader_vrf_input(era, slot, eta0)`, checks full-enum equality against
  `answer.expected_vrf_input` (catches wrong-era AND wrong-bytes → `VrfInputMismatch`),
  verifies over `expected.alpha_bytes()`.
- `is_leader_for_vrf_output` applies `leader_value_for(&answer.expected_vrf_input, output)`
  before the threshold (Praos → `praos_leader_value`).

### RED — `ade_node::produce_mode::run_real_forge`
- Step 1 proves over `ctx.leader_schedule_answer.expected_vrf_input.alpha_bytes()`
  (the single authority's bytes) — no independent `vrf_input(...)` (N3).
- Step 2 passes the era (from the S1 guard) into `verify_and_evaluate_leader`.
- Drop the now-unused `vrf_input`/`VrfRole` import.

### Fixture sites (mechanical, `LeaderScheduleAnswer` retype)
`forge.rs`, `fixtures.rs`, `scheduler.rs`, `tick_assembler.rs`,
`producer_pipeline_slot_deadline.rs`, `forge_handler_variants.rs`,
`forge_succeeds.rs`, `leader_check.rs` tests → `ExpectedVrfInput::Praos(...)` /
`leader_vrf_input(Conway, …)`.
