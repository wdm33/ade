# Slice S4 — cardano-cli ground-truth corpus + compatibility CI gate

**Cluster:** PHASE4-N-P
**Strengthens:** I1, I2 (Haskell `cardano-base` ground-truth); N6
(throwaway-comment-prefixed hex-literal corpus); N9 (mechanical
gate against upstream-shim reintroduction).
**Introduces:** `crates/ade_crypto/src/kes_sum/cardano_cli_corpus.rs`
(test-only); `ci/ci_check_kes_sum_compatibility.sh`.
**Dependencies:** S1 (proof obligation), S2 (BLUE algorithm), S3
(serde).

## §1 Intent

Cross-impl validation against the **real cardano-cli output** that
`cardano-cli node key-gen-KES` emits, captured from the docker
preprod peer (`cardano-cli 11.0.0.0`, `ghc-9.6`). Three throwaway
SKEY + VKEY pairs are committed as hex-literal `&[u8; N]`
constants — never as `.skey` envelope files — and exercised
through the full serde + sign/verify pipeline.

This slice closed the most important compatibility finding of
PHASE4-N-P:

### Discovery: Haskell vs cardano-crypto Rust 1.0.8 prefix divergence

When the S4 fixtures were first written against the S3 implementation,
the deserializer rejected all three pairs with
`KesParseError::InconsistentSubtreeVkRight { level: 2 }`. Root cause:

| Source | Seed-expansion prefixes |
|---|---|
| Haskell `cardano-base` `expandSeed` (the source of cardano-cli output) | `0x01` (left), `0x02` (right) |
| `cardano-crypto` 1.0.8 Rust `KesHashAlgorithm::expand_seed` | `0x00` (left), `0x01` (right) |

S2's cross-impl tests against `cardano-crypto` Rust passed because
ade's S2 impl also used `0x00`/`0x01` — but BOTH were divergent from
Haskell. The cardano-cli corpus established Haskell as the ground
truth; `crates/ade_crypto/src/kes_sum/hash.rs` was corrected to
`0x01`/`0x02`. The S2 cross-impl-vs-upstream tests were repurposed
to *document the divergence* rather than assert agreement.

DC-CRYPTO-08 invariant statement (the registry entry written in
the `/invariants` step) said "byte-identical to Haskell
`cardano-base`" — that wording stands. The discovery confirms the
invariant was specified correctly; the implementation now matches
it. Without the cardano-cli ground-truth corpus we would have
shipped an algorithm that internally validates against
`cardano-crypto` Rust but fails to interoperate with actual
cardano-node output.

This is a textbook
[[feedback-real-interop-finds-codec-bugs]] outcome: synthetic
round-trip tests miss wire-format bugs; validate against real
output before claiming the codec closed.

## §2 Scope

### `crates/ade_crypto/src/kes_sum/cardano_cli_corpus.rs` (new, `#[cfg(test)]`)

Three throwaway-key fixtures, each a `&[u8; 608]` SKEY + a
`&[u8; 32]` VKEY constant, prefixed with the mandatory throwaway
comment:

```text
// TEST ONLY: throwaway deterministic fixture generated for Sum6KES
// equivalence testing. Not a Cardano node hot key. Never use on any network.
```

Aggregator `ALL_PAIRS: &[(&[u8; 608], &[u8; 32])]` for the
cross-impl tests.

### Test additions (`kes_sum/tests.rs`)

- `cardano_cli_corpus_skey_deserializes_and_vk_matches_ground_truth` —
  each SKEY round-trips through our deserializer; derived VK matches
  the captured VKEY byte-for-byte; current_period is 0.
- `cardano_cli_corpus_skey_round_trips_through_ade_serde` —
  parse(SKEY) → serialize → byte-identical.
- `cardano_cli_corpus_skey_period_inference_is_zero` —
  `period_from_zeroed_sum6_tree_shape` returns Ok(0) for every
  fresh-from-keygen SKEY.
- `cardano_cli_corpus_cross_impl_vk_matches_cardano_crypto` — VK
  match via Haskell ground truth (our impl == cardano-cli; the
  divergence vs `cardano-crypto` Rust is documented separately).
- `cardano_cli_corpus_sign_then_upstream_verifies` — ade signs at
  period 0 using a corpus-deserialized key; cardano-crypto Rust
  verifies the signature against the captured VKEY. Works because
  `verify_kes` only walks the hash chain + Ed25519 leaf check —
  neither depends on `expand_seed`.
- `cardano_cli_corpus_negative_flip_one_byte_in_vk_left_fail_closed` —
  adversarial: flip byte 544 of SKEY1 (level-6 vk_left), expect
  `KesParseError::InconsistentSubtreeVkLeft { level: 6 }`.
- `cardano_cli_corpus_constants_have_expected_sizes`.

Plus, from the S4 fix:

- `sum6_kes_seed_expansion_diverges_from_cardano_crypto_rust_1_0_8` —
  asserts our `expand_seed` ≠ upstream's (documents the
  divergence; flips to failure if a future upstream version aligns
  to Haskell, prompting a hash.rs comment update).
- `sum6_kes_vk_diverges_from_cardano_crypto_rust_for_same_seed` —
  same divergence at the VK level.

### `ci/ci_check_kes_sum_compatibility.sh` (new)

4 mechanical guards:

1. **Corpus exists + throwaway comments.** Each SKEY{N} const
   must be preceded by the throwaway comment. Min 3 fixtures.
2. **No `.skey` envelope files** under `crates/ade_crypto/`.
3. **`cardano_crypto::kes` only inside `#[cfg(test)]`** within
   `crates/ade_crypto/src/**`. Skips test-only files
   (`tests.rs`, `*_tests.rs`, `*_corpus.rs`). N9 is KES-scoped;
   VRF and DSIGN imports continue to be allowed in production
   (separate future-cluster concerns).
4. **`expand_seed` prefix bytes match Haskell** — literal-grep
   that `hash.rs` sets `left_input[0] = 0x01` and
   `right_input[0] = 0x02`, and explicitly rejects the
   `cardano-crypto` Rust 1.0.8 prefixes (0x00 / 0x01).

### S2 cross-impl-vs-upstream tests retired

The S2 cross-impl tests against `cardano-crypto` Rust 1.0.8
(`sum6_kes_cross_impl_vk_matches_cardano_crypto`,
`sum6_kes_cross_impl_seed_expansion_matches_cardano_crypto`, the
two sign/verify direction tests) were premised on a false
equivalence and have been deleted. Their replacement is the
cardano-cli corpus + the divergence-documenting tests above.
