# Slice S1 — `period_from_zeroed_sum6_tree_shape` proof obligation

**Cluster:** PHASE4-N-P
**Strengthens:** I5 (period uniquely inferable), N3 (no
best-effort guesswork), N4 (no period overflow).
**Introduces:** no code; no registry entries (DC-CRYPTO-08 and
DC-CRYPTO-09 already exist as `status = "declared"`, appended
during the `/invariants` step).
**Dependencies:** PHASE4-N-P invariants doc
([`docs/planning/phase4-n-p-invariants.md`](../../planning/phase4-n-p-invariants.md))
+ cluster doc ([`cluster.md`](cluster.md)). No code-slice
predecessor.

## §1 Intent

Write the deserialization algorithm down — with byte offsets,
pseudocode, induction proof, closed `KesParseError` variants,
and the exhaustive 64-period fixture-test plan — **before any
deserializer code lands**. Per [[feedback-proof-discipline]], the
period-from-zeroed-tree-shape recurrence is exactly the kind of
"unknown compatibility fact" that must be a slice-entry proof
obligation, not a footnote on the implementing slice.

S3 (`raw_deserialize_signing_key_kes` + period inference) cannot
start until this doc is committed. The doc is the spec; S3 is
the implementation of the spec.

## §2 Scope

### Deliverable

One file:
[`period-from-zeroed-sum6-tree-shape-proof.md`](period-from-zeroed-sum6-tree-shape-proof.md)

It MUST contain:

1. The Sum_n recursive tree layout, with explicit byte offsets for
   each of the 19 fields in the 608-byte payload (leaf sk + 6
   levels × 3 fields).
2. Forward-secrecy semantics: what `update_kes` mutates at the
   level-`n` boundary, why the seed is zeroed, what is recoverable
   after that point and what isn't.
3. Pseudocode for `period_from_zeroed_sum6_tree_shape(&[u8; 608]) -> Result<u32, KesParseError>`
   with the leaf-zero guard + per-level seed-zero check.
4. An induction-on-depth proof sketch that the function agrees
   with the `update_kes` chain for every `p ∈ 0..=63`.
5. The closed `KesParseError` enum — exactly the variants S3 may
   add, with the discipline that they MUST carry only
   non-secret metadata (level number, payload size — never raw
   key bytes / seeds / hex / decimal representations of secret
   material).
6. The 64-period exhaustive fixture-test plan — the test names
   S3 must add and their assertion shapes.
7. The verification-chain walk that S3 must implement after
   period inference (per-level vk_left / vk_right reconstruction
   when verifiable; trust-as-given when forward-secrecy makes
   re-derivation impossible).
8. Out-of-scope concerns (mlock, constant-time leaf check, CBOR
   envelope shape).

### Not in scope (S3+)

- Implementing the function in Rust.
- Adding `KesParseError` to the codebase.
- Writing the 64-period test (just listing what it must assert).
- Building the test corpus.

S1 is purely documentary. The proof obligation document is the
artifact; no `.rs` file is touched.

## §3 Mechanical acceptance criteria

1. `docs/clusters/PHASE4-N-P/period-from-zeroed-sum6-tree-shape-proof.md`
   exists at the cluster directory.
2. The doc's §2 (on-disk byte layout) lists all 19 fields with
   correct byte offsets (sum = 608).
3. The doc's §3 (pseudocode) includes the leaf-zero guard and 6
   per-level seed-zero checks with contributions 32, 16, 8, 4,
   2, 1 (descending levels 6..=1).
4. The doc's §4 (induction proof) covers both branches at the
   inductive step (left sub-tree active / right sub-tree active).
5. The doc's §5 lists exactly the `KesParseError` variants
   S3 will introduce: `WrongPayloadSize { actual }`,
   `LeafSignKeyAllZero`,
   `InconsistentSubtreeVkLeft { level }`,
   `InconsistentSubtreeVkRight { level }`,
   `LevelOutOfRange { level }`. No private-key bytes carried by
   any variant.
6. The doc's §6 names the 3 test functions S3 must add
   (`period_from_zeroed_sum6_tree_shape_agrees_with_update_kes_chain`,
   `round_trip_preserves_period_for_every_period`,
   `wrong_payload_size_rejected`,
   `leaf_all_zero_rejected`,
   `inconsistent_vk_left_rejected_when_verifiable`) and their
   shapes.
7. The doc's §7 describes the verification walk that S3 must
   implement (leaf → root traversal with per-level vk checks).
8. The doc cross-references the invariants doc (§1.I5, §2.N3,
   §2.N4) and the Haskell + cardano-crypto Rust sources.
9. No code committed. Slice is documentation-only.
10. Commit message in IDD style with the project-override
    trailer.

## §4 Hard prohibitions

- **No code in this slice.** S1 is purely documentary; any
  `.rs` change is out of scope.
- **No heuristic period inference language** in the doc. The
  algorithm returns exactly one `u32` or a closed
  `KesParseError`. No "best matching tree", no
  "closest valid period", no fallback inference rules.
- **No raw key bytes in any example error variant payload.**
  Per N6 (invariants doc), variants carry only `level: u32` and
  `actual: usize`. The doc must show this discipline explicitly.
- **No silent acceptance of period > 63.** The doc must show
  that Sum6 encodes period 0..=63 by structure and that S3's
  deserializer carries this discipline.

## §5 Replay obligations preserved

S1 introduces no new replay obligations. The replay-equivalence
claim (I3 round-trip identity, I5 unique period inference) is
the *target* of the proof obligation; S3 will mechanically
enforce it.
