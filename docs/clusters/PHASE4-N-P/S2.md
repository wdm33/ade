# Slice S2 — Ade-owned BLUE Sum6KES algorithm

**Cluster:** PHASE4-N-P
**Strengthens:** I1, I2, I4, I6 (BLUE algorithm equivalence,
cross-impl agreement, `update_kes` chain, forward secrecy),
D1–D4 (deterministic surface), N1 (no cross-impl divergence),
N7 (Drop zeroize), N9 (no upstream-shim in production code that
S2 introduces).
**Introduces:** new BLUE module `crates/ade_crypto/src/kes_sum/`.
DC-CRYPTO-08 stays `declared`; flips to `enforced` on S5 close.
**Dependencies:** PHASE4-N-P S1 (proof obligation doc committed
at HEAD `c04dc0b`).

## §1 Intent

Implement Sum0KES (`SingleKes<Ed25519>`) and the generic
recursive `SumKes<D>` from first principles, matching the Haskell
`cardano-base` algorithm and the upstream `cardano-crypto` 1.0.8
Rust reference byte-for-byte. Ship the algorithm only —
**no expanded-skey serde** (deferred to S3, which the period
proof obligation gates).

After S2:
- `ade_crypto::kes_sum::Sum6Kes` exists with `gen_key_kes_from_seed_bytes`,
  `derive_verification_key`, `update_kes`, `sign_kes`,
  `verify_kes`.
- Cross-impl equivalence with `cardano-crypto` 1.0.8 is
  mechanically checked over the 64-period chain + a 4-message
  corpus.
- `Drop` for each new signing-key type best-effort zeroizes
  seeds and any owned secret buffers.
- `cardano-crypto` continues to be referenced by `ade_crypto`'s
  production code under `kes::verify_kes_signature` — that
  migrates in S3 when sig deserialization lands. S2 does not
  introduce a new production import of `cardano-crypto`; cross-
  impl test code under `#[cfg(test)]` imports it via the existing
  dependency declaration.

## §2 Scope

### New BLUE module

`crates/ade_crypto/src/kes_sum/`:

- `mod.rs` — public surface: `KesAlgorithm` trait, `Sum0Kes`,
  `SumKes<D>`, type aliases `Sum1Kes`..`Sum6Kes`, `KesError`.
- `single.rs` — `Sum0Kes` impl (wraps `ed25519-dalek` directly;
  no upstream KES path).
- `sum.rs` — generic `SumKes<D>` impl with the recursive sum
  construction.
- `hash.rs` — domain-separated seed expansion
  (`expand_seed(seed) -> (left_seed, right_seed)`) using
  `Blake2b256` from `crate::blake2b`. Matches upstream's
  `0x00 || seed` / `0x01 || seed` prefix byte convention.
- `tests.rs` (`#[cfg(test)]`) — 64-period chain test +
  cross-impl agreement tests against `cardano_crypto::kes::Sum6Kes`.

### Public surface (`kes_sum/mod.rs`)

```rust
pub trait KesAlgorithm {
    type VerificationKey: Clone + Eq + Sized;
    type SigningKey;
    type Signature: Clone + Eq;

    const ALGORITHM_NAME: &'static str;
    const SEED_SIZE: usize;
    const VERIFICATION_KEY_SIZE: usize;
    const SIGNING_KEY_SIZE: usize;
    const SIGNATURE_SIZE: usize;

    fn total_periods() -> u32;

    fn gen_key_kes_from_seed_bytes(seed: &[u8]) -> Result<Self::SigningKey, KesError>;
    fn derive_verification_key(sk: &Self::SigningKey) -> Self::VerificationKey;
    fn update_kes(sk: Self::SigningKey, period: u32)
        -> Result<Option<Self::SigningKey>, KesError>;
    fn sign_kes(sk: &Self::SigningKey, period: u32, msg: &[u8])
        -> Result<Self::Signature, KesError>;
    fn verify_kes(vk: &Self::VerificationKey, period: u32, msg: &[u8],
                  sig: &Self::Signature) -> Result<(), KesError>;
}

pub enum KesError {
    InvalidSeedLength { expected: usize, actual: usize },
    PeriodOutOfRange { period: u32, max_period: u32 },
    VerificationFailed,
    KeyExpired,
    Ed25519(&'static str),  // wraps ed25519-dalek errors; static detail only
}

pub struct Sum0Kes;
impl KesAlgorithm for Sum0Kes { ... }

pub struct SumKes<D> { _phantom: PhantomData<D> }
impl<D: KesAlgorithm> KesAlgorithm for SumKes<D> { ... }

pub type Sum1Kes = SumKes<Sum0Kes>;
pub type Sum2Kes = SumKes<Sum1Kes>;
pub type Sum3Kes = SumKes<Sum2Kes>;
pub type Sum4Kes = SumKes<Sum3Kes>;
pub type Sum5Kes = SumKes<Sum4Kes>;
pub type Sum6Kes = SumKes<Sum5Kes>;
```

### Signing key + signature representations

```rust
pub struct Sum0SigningKey {
    seed: [u8; 32],   // Ed25519 seed
}

pub struct SumSigningKey<D: KesAlgorithm> {
    sk_child: D::SigningKey,
    r1_seed:  Option<[u8; 32]>,
    vk0:      [u8; 32],
    vk1:      [u8; 32],
}

pub struct Sum0Signature {
    bytes: [u8; 64],  // Ed25519 signature
}

pub struct SumSignature<D: KesAlgorithm> {
    sigma: D::Signature,
    vk0:   [u8; 32],
    vk1:   [u8; 32],
}
```

`VerificationKey` is `[u8; 32]` for both `Sum0Kes` (ed25519 pk)
and every `SumKes<D>` (Blake2b256 hash of `vk0 || vk1`).

### Drop discipline

- `Sum0SigningKey::drop` — `seed.iter_mut().for_each(|b| *b = 0)`
  + `core::hint::black_box`.
- `SumSigningKey<D>::drop` — if `r1_seed.is_some()`, zeroize the
  inner 32 bytes. Inner `sk_child` is dropped recursively; its
  own `Drop` zeroizes recursively.

Custom `Debug` impls for both: print only struct name +
`"<redacted>"`. No field-by-field dump.

### Seed expansion

```rust
pub fn expand_seed(seed: &[u8; 32]) -> ([u8; 32], [u8; 32]) {
    let mut left_input  = Vec::with_capacity(33);
    left_input.push(0x00);
    left_input.extend_from_slice(seed);
    let left = ade_crypto::Blake2b256::hash(&left_input);

    let mut right_input = Vec::with_capacity(33);
    right_input.push(0x01);
    right_input.extend_from_slice(seed);
    let right = ade_crypto::Blake2b256::hash(&right_input);

    (left, right)
}
```

The `0x00 / 0x01` prefix bytes match upstream's seed-expansion
domain separation. Tested against upstream's `H::expand_seed` in
the cross-impl harness.

## §4 Hard prohibitions

- **N9.** No production code in `crates/ade_crypto/src/kes_sum/`
  imports `cardano_crypto`. `#[cfg(test)]` modules may import for
  cross-impl validation.
- **N7.** Every signing-key type hand-rolls `Drop` to zeroize
  seed buffers.
- No public byte accessors on `Sum0SigningKey` / `SumSigningKey`.
- No `Clone` derive on signing-key types (forces explicit
  custody-aware handling).
- No `pub` on the inner fields of signing-key / signature types.
