# Slice S1 — Ade-native KES envelope + key-gen + cardano-cli fail-closed

**Cluster:** PHASE4-N-O
**Strengthens:** OP-OPS-04 (Ade-native flow), T-KEY-01.
**Introduces:** DC-CRYPTO-06 (closed envelope format),
DC-CRYPTO-07 (cardano-cli expanded fail-closed).
**Dependencies:** PHASE4-N-C (producer pipeline + KES signing
in `ade_runtime::producer::signing` ships intact).

## §1 Intent

Close the **Ade-native KES operator flow** half of
OP-OPS-04 from PHASE4-N-C:

- Ade owns its hot-KES seed format. The on-disk shape is a
  closed JSON envelope `ade.kes.seed.v1` carrying
  `{format, role, crypto, seed_32, period_idx, format_version}`.
- `ade_node key-gen-KES --out-file PATH` is the only Ade
  surface that produces this envelope. Entropy sourced from
  `/dev/urandom` (or `--seed-file PATH` for testability).
- `load_ade_kes_signing_key(PATH)` is the only loader that
  accepts the envelope and constructs a `KesSecret`.
- cardano-cli's `KesSigningKey_ed25519_kes_2^6` envelope — any
  payload shape (32-byte seed, 608-byte expanded, 612-byte
  malformed, anything) — fail-closes with
  `KeyLoadError::UnsupportedExpandedKesKeyFormat`. No fallback
  parser.
- Closed CLI success vocabulary: exactly four lines (filename,
  format, role, VK fingerprint). No seed/key bytes anywhere.

After this slice OP-OPS-04's `open_obligation` is cleared for
the Ade-native flow; the cardano-cli-import obligation is
re-emitted as
`blocked_until_phase4_n_p_kes_sum_reimpl`.

## §2 Scope

### BLUE/GREEN code changes

None. KES signing math stays in `ade_runtime::producer::signing`

### RED code changes

#### `crates/ade_runtime/src/producer/ade_kes_envelope.rs` (new)

Closed JSON-envelope parser/writer. Pure functions only:

```rust
pub const ADE_KES_ENVELOPE_FORMAT: &str = "ade.kes.seed.v1";
pub const ADE_KES_ROLE: &str = "kes_hot_signing_key";
pub const ADE_KES_CRYPTO: &str = "Sum6KES-Ed25519DSIGN";
pub const ADE_KES_FORMAT_VERSION: u32 = 1;

#[derive(Debug, PartialEq, Eq)]
pub struct AdeKesEnvelope {
    pub seed_32: [u8; 32],
    pub period_idx: u32,
}

#[derive(Debug, PartialEq, Eq)]
pub enum AdeKesEnvelopeError {
    UnknownEnvelopeFormat,
    WrongKeyRole,
    UnsupportedCryptoTag,
    MissingSeed32,
    MalformedSeed32 { detail: &'static str },
    MalformedPeriodIdx,
    UnsupportedFormatVersion,
    MalformedJson { detail: &'static str },
}

pub fn parse(buf: &[u8]) -> Result<AdeKesEnvelope, AdeKesEnvelopeError>;
pub fn serialize(envelope: &AdeKesEnvelope) -> Vec<u8>;
```

- Internal struct `EnvelopeOnDisk` uses serde with
  `#[serde(deny_unknown_fields)]` on the LOAD-BEARING fields
  (`format`, `role`, `crypto`, `seed_32`, `period_idx`,
  `format_version`). Optional metadata fields
  (`genesis_hash`, `network_magic`, `created_at_slot`,
  `created_by`) are collected into a sidecar
  `MetadataOnDisk` that the loader ignores — N-O does not
  mechanically enforce them; PHASE4-N-O-METADATA (if ever
  needed) would.
- `seed_32` field is hex-encoded (lower-case, exactly 64 hex
  chars).
- `period_idx` is a JSON integer ≤ `SUM6_MAX_PERIOD` (64) at
  parse time.
- `serialize` writes keys in a fixed order matching the user
  spec verbatim: `format`, `role`, `crypto`, `seed_32`,
  `period_idx`, `format_version`.

```rust
pub fn load_ade_kes_signing_key(path: &Path) -> Result<KesSecret, KeyLoadError> {
    let mut buf = Vec::new();
    File::open(path).map_err(|e| KeyLoadError::Io(e.kind()))?
        .read_to_end(&mut buf).map_err(|e| KeyLoadError::Io(e.kind()))?;
    let env = ade_kes_envelope::parse(&buf).map_err(KeyLoadError::AdeEnvelope)?;
    KesSecret::from_seed_at_period(&env.seed_32, env.period_idx)
        .map_err(KeyLoadError::Crypto)
}
```

#### `crates/ade_node/src/key_gen.rs` (new)

```rust
pub async fn run_key_gen_kes(cli: KeyGenKesCli) -> ExitCode { ... }
```

Behavior:
1. Source 32 bytes of entropy: read `cli.seed_file` if Some,
   else open `/dev/urandom` and read 32 bytes. On I/O error
   print one closed line and exit `EXIT_GENERIC_STARTUP`.
2. Call `write_ade_kes_envelope(&cli.out_file, &seed,
   cli.period_idx.unwrap_or(0))`. On error map to
   `EXIT_GENERIC_STARTUP`.
3. Reload via `load_ade_kes_signing_key(&cli.out_file)` —
   round-trip is a self-check; mismatch is a hard
   fail-closed (`EXIT_GENERIC_STARTUP`).
4. Compute VK fingerprint via
   `kes_secret.verification_key_fingerprint()`.
5. Print exactly four lines to stdout:

   ```
   Generated Ade KES key: <out_file>
   Format: ade.kes.seed.v1
   Role: kes_hot_signing_key
   Public verification key fingerprint: <hex>
   ```

6. Zeroize the local seed copy + drop the `KesSecret`. Return
   `ExitCode::SUCCESS`.

No JSONL writer touched. No live log emitted. Key-gen is a
one-shot operation; the LiveLogWriter / AdmissionLogWriter
remain unused.
