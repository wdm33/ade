## Conventions

- A **module** in Ade is a Cargo workspace crate (smallest independently-buildable unit). Source: `Cargo.toml` `[workspace] members` (**12** entries). One exception to crate-granularity: `ade_network` is split by *submodule color* — `.idd-config.json` `core_paths` resolves BLUE at the submodule path level (9 paths), so its BLUE / GREEN / RED submodules are documented as separate entries.

### TCB color sources (in order of authority)

1. `.idd-config.json` `core_paths` — substring match against the module's absolute path ⇒ BLUE (6 BLUE crates: `ade_ledger`, `ade_codec`, `ade_types`, `ade_crypto`, `ade_plutus`, `ade_core`; plus 9 BLUE `ade_network` submodule paths).
2. `CLAUDE.md` naming/role notes (`ade_runtime` RED shell; `ade_testkit` GREEN test infra; `ade_node` the binary).
3. Module banner signals — the `// Core Contract:` / `//! GREEN …` / `// RED …` / `// … BLUE` headers at the top of each source file, plus the BLUE `lib.rs` deny attributes.
4. Import heuristic — a module reaching for tokio / sockets / files / clock / rand is RED; a deterministic glue module over BLUE is GREEN.

## Cross-Module Rules (project-wide)

### Dependency direction (RED → GREEN → BLUE; never outward)

Verified at HEAD from each crate's `[dependencies]`:

- **BLUE** crates depend only inward: `ade_types` (leaf, no deps) ← `ade_codec` / `ade_crypto` (only `ade_types`) ← `ade_core` (`ade_types`, `ade_crypto`) ← `ade_plutus` (`ade_types`, `ade_crypto`, `ade_codec`) ← `ade_ledger` (`ade_types`, `ade_crypto`, `ade_codec`, `ade_plutus`, `ade_core`). `ade_network` BLUE submodules depend only on `ade_codec` + `ade_types`.
- **GREEN** (`ade_testkit`, and GREEN-by-content sub-trees inside RED crates) depend on BLUE; must not affect authoritative outputs.
- **RED** (`ade_runtime`, `ade_node`, `ade_core_interop`, `ade_network::mux::transport`, `ade_mem_diag`) depend on BLUE/GREEN. `ade_node` depends on `ade_runtime` + `ade_mem_diag` (binary-only, behind the toggle); `ade_core_interop` is a RED leaf (nothing depends on it except `ade_testkit` dev/plain). **No BLUE crate depends on any RED/GREEN crate.** The native Mithril surfaces obey this: the BLUE decoders (`mithril_utxo_materialize` / `ledgerdb_state` / `ledgerdb_tables`) live in `ade_ledger` and are consumed by the RED `ade_runtime::mithril_native_assembly` + `ade_node::native_firstrun`, never the reverse. The ECA activation reads the durable ChainDB from RED `ade_node`/`ade_runtime` and consumes the BLUE `EpochConsensusView` THROUGH `ade_core::consensus::ledger_view`.

Every crate is `ade_*`. Color is NOT encoded in the crate name (unlike ziranity-v3's `ion_*`/`ziranity_*`); it is resolved by `.idd-config.json` `core_paths` (BLUE) + the module banners. `ade_network` is the one crate split by submodule color.
