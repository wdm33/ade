## Conventions

- A **module** in Ade is a Cargo workspace crate (smallest independently-buildable unit). Source: `Cargo.toml` `[workspace] members` (11 entries). One exception to crate-granularity: `ade_network` is split by *submodule color* — `.idd-config.json` `core_paths` resolves BLUE at the submodule path level (9 paths), so its BLUE / GREEN / RED submodules are documented as separate entries.
- Several BLUE crates host sub-trees that carry a *different* TCB color by their own module banner and cluster TCB Color Map; and the RED `ade_runtime` crate hosts numerous **GREEN-by-content** sub-trees. These are surfaced as sub-classification notes inside the owning crate's entry (or, where load-bearing, as their own GREEN/RED entries below).
- Modules are listed by TCB color (BLUE → GREEN → RED), alphabetical within each color.

### TCB color sources (in order of authority)

1. `.idd-config.json` `core_paths` — substring match against absolute path. BLUE crate prefixes: `ade_codec`, `ade_types`, `ade_crypto`, `ade_core`, `ade_ledger`, `ade_plutus`. Plus the 9 BLUE `ade_network` submodule paths: `mux/frame.rs`, `codec/`, `handshake/`, `chain_sync/`, `block_fetch/`, `tx_submission/`, `keep_alive/`, `peer_sharing/`, `n2c/`.
2. `.idd-config.json` `_core_paths_doc` — `ade_runtime` is RED; `ade_testkit` is GREEN; `ade_node` is RED; `ade_network::mux::transport` + `ade_network::session` are nominally RED (but `session/` is GREEN-by-content per PHASE4-N-L). `ade_core_interop` is RED.
3. **Module doc-comment banners** — every BLUE/GREEN source file opens with the `// Core Contract:` banner followed by `//! BLUE …` / `//! GREEN …` / `//! RED …`. This is the per-file authority that distinguishes GREEN-by-content from RED inside `ade_runtime`. Verified directly against the tree at HEAD for every new/changed module in this regeneration.
4. Cluster TCB Color Maps under `docs/clusters/` and `docs/clusters/completed/` — §5 "TCB color map (FC/IS partition)" of each `cluster.md`.
5. The invariant registry `docs/ade-invariant-registry.toml` (303 rules) — each rule's `code_locus` + `statement` is the authoritative source for the **MUST NOT** rows below.

### Active cluster at HEAD

**PHASE4-N-F-A — CLOSED (recovered seed-epoch consensus-input CAPABILITY).** HEAD `a3a0636` is the close-tail commit (an A5-scoping doc revision following the A1–A4 implementation). The cluster establishes the **recovered seed-epoch consensus inputs** as a first-class Ade-owned recovered surface — the closed `SeedEpochConsensusInputs` canonical record (PoolDistr, ASC, per-pool VRF keyhash, total active stake) for the single seed `epoch_no`, persisted as a fingerprint-bound sidecar and reconstructable through verified warm-start. It is a **capability cluster, not production wiring**: the authority surface is proven end-to-end, but no production mode is wired to the recovery path.

**Critical scope discipline (do not overclaim — every entry below honors these):**
- **A4's projection is a capability.** `PoolDistrView::from_seed_epoch_consensus_inputs` is a pure BLUE field-map that proves the recovered surface is a drop-in leadership source (DC-CINPUT-02a). The producer (`ade_node::produce_mode`) does **not** consume it — `produce_mode` still cold-starts from `--consensus-inputs-path`, projects the operator bundle via the private `pool_distr_view_from_consensus_inputs`, and passes `SeedEpochConsensusSource::NotRequired`. Producer consumption (CE-A-4b) is deferred to a successor cluster (PHASE4-N-F-C).
- **A3b's warm-start verification is proven at the authority surface only.** `bootstrap_initial_state(RequiredFromRecoveredProvenance)` restores + verifies the sidecar fail-closed, exercised directly. No production restart path is wired: `node.rs run_node_until_shutdown` is the binary's main loop but the recovery-threading path is not built, and `ade_runtime::recovery::restart::recover_node_state` is the test-only secondary recovery entry. `produce_mode` cold-starts. DC-CINPUT-01 carries this as its open obligation (deferred to PHASE4-N-F-C / C3).
- **BA-02 is NOT satisfied anywhere.** No Haskell peer accepts an Ade-forged block at this HEAD; the bounty's block-acceptance leg remains gated on the producer recovered-state lifecycle (a named successor cluster).

The cluster introduces / extends:
- **BLUE** `ade_ledger::seed_consensus_inputs` (NEW, A1) — the closed `SeedEpochConsensusInputs` record + its SOLE version-gated, byte-canonical codec (`encode_/decode_seed_epoch_consensus_inputs`, `SEED_CINPUT_SCHEMA_VERSION = 1`) + closed `SeedConsensusInputsError`.
