## Conventions

- A **module** in Ade is a Cargo workspace crate (smallest independently-buildable unit). Source: `Cargo.toml` `[workspace] members` (11 entries). One exception to crate-granularity: `ade_network` is split by *submodule color* — `.idd-config.json` `core_paths` resolves BLUE at the submodule path level (9 paths), so its BLUE / GREEN / RED submodules are documented as separate entries.
- Several BLUE crates host sub-trees that carry a *different* TCB color by their own module banner; the RED `ade_runtime` and `ade_node` crates host numerous **GREEN-by-content** sub-trees. These are surfaced as sub-classification notes inside the owning crate's entry (or, where load-bearing, as their own GREEN/RED entries below). **MEM-MEASURE / MEM-OPT-OPS added the `ade_node::mem_measure` sub-tree** — three file-separated TCB colors inside the RED `ade_node` crate (RED `rss_sampler`, GREEN `bounded_admission` / `evidence`, GREEN/RED `runner`); it is documented as its own RED-host entry below. The PHASE4-N-AO `ade_node` SELECT modules (`candidate_aggregator` / `lca_walk` / `selector_state` / `post_switch_continuity` / `fork_switch` / `fair_merge`) are carried verbatim from the prior refresh.
- Modules are listed by TCB color (BLUE → GREEN → RED), alphabetical within each color.
